# JNI-008 漏洞详细分析报告

## 1. 漏洞概述

### 基本信息
| 属性 | 值 |
|------|-----|
| **漏洞 ID** | JNI-008 |
| **类型** | Stack-based Buffer Overflow (CWE-121) |
| **严重性** | High |
| **置信度** | 85/100 |
| **状态** | CONFIRMED |
| **文件路径** | `cpp-omni/src/jni/deserializer.cpp` |
| **行号** | 174-178 |
| **函数名** | `Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_rowShuffleParseBatch` |
| **信任边界** | JNI Interface (High Risk) - Spark JVM Process → Native C++ Library |

### 漏洞摘要
JNI 函数 `rowShuffleParseBatch` 中使用来自 protobuf 的未验证值 `vecCount` 创建 VLA (Variable Length Array) `vecs[vecCount]`。由于 `vecCount` 来自外部不可信数据源（protobuf Shuffle 数据），攻击者可通过构造恶意 protobuf 数据指定超大 `vecCount` 值，导致栈溢出、内存分配失败或进程崩溃。

### 关联漏洞
该漏洞与以下漏洞具有相同的根因模式（VLA 栈分配 + 未验证的 protobuf 值）：
- **JNI-007** (`columnarShuffleParseBatch:74-77`): 相同的 VLA 栈溢出模式
- **JNI-003** (`rowShuffleParseInit:127-136`): ParseFromArray 返回值未检查，导致无效 protobuf 对象
- **JNI-015** (`rowShuffleParseBatch:200-206`): 同一函数中 offsets 数组越界访问

### 2.2 VLA (Variable Length Array) 安全问题分析

#### VLA 在 C/C++ 中的行为

```cpp
// VLA 是 C99 特性，在某些编译器中支持（如 GCC、Clang）
int32_t vecCount = 1000000;  // 来自 protobuf
omniruntime::vec::BaseVector* vecs[vecCount];  // 栈上分配
```

**栈分配机制**：
- VLA 在函数栈帧上分配，大小由运行时值决定
- 每个指针 8 字节（64 位系统），`vecs[vecCount]` 需要 `vecCount * 8` 字节栈空间
- 典型线程栈大小：8MB (Linux) 或 1MB (某些 JVM 配置)

**危险计算**：
| vecCount 值 | 栈需求 | 结果 |
|-------------|--------|------|
| 1000 | 8KB | 正常 |
| 100,000 | 800KB | 可能正常 |
| 1,000,000 | 8MB | 接近栈上限，风险 |
| 10,000,000 | 80MB | 栈溢出崩溃 |
| 100,000,000 | 800MB | 确定栈溢出 |
| INT_MAX (2,147,483,647) | ~17GB | 理论不可能，实际触发立即崩溃 |

#### VLA 的替代方案

安全做法是使用 `std::vector`（堆分配）：
```cpp
// 安全版本
std::vector<omniruntime::vec::BaseVector*> vecs(vecCount);  // 堆分配
```

### 2.4 缺失的验证检查

当前代码中**完全没有**以下验证：

1. **vecCount 范围检查**：无上限检查，接受任意 int32_t 值
2. **vecTypes.size() 与 vecCount 一致性检查**：无验证，可能导致越界访问
3. **Java 数组长度与 vecCount 匹配检查**：无验证，可能导致 Java 数组越界写入
4. **rowCount 范围检查**：无上限检查
5. **rows buffer 与 offsets 一致性检查**：无验证

### 2.5 Java 端数组分配分析

```java
// ShuffleDataSerializer.java:68-75
int vecCount = deserializer.getVecCount();  // 来自 JNI，基于 protobuf
int rowCount = deserializer.getRowCount();

// Java 端基于 vecCount 创建数组
int[] typeIdArray = new int[vecCount];
int[] precisionArray = new int[vecCount];
int[] scaleArray = new int[vecCount];
long[] vecNativeIdArray = new long[vecCount];

deserializer.parse(typeIdArray, ...);  // JNI 调用
```

**问题**：
- Java 端的数组大小也基于来自 JNI 的 `vecCount`
- 如果 `vecCount` 超大（如 10,000,000），Java 端也会尝试分配超大数组
- Java 堆可能抛出 `OutOfMemoryError`
- 但在 C++ 端，栈溢出会在 Java 数组分配之前就发生

### 3.3 攻击场景分析

#### 场景 1: 恶意 Executor 攻击

**前提条件**：
- Spark 集群多租户环境
- 攻击者可提交恶意 Spark 任务或控制某个 Executor

**攻击步骤**：
1. 在恶意 Executor 的 Map 任务中，构造包含超大 `vecCnt` 的 protobuf Shuffle 数据
2. 数据通过 Shuffle 机制传输到其他 Executor
3. 目标 Executor 在 Reduce 任务中反序列化该数据
4. 调用链：`deserialize` → `init` → `parse` → `rowShuffleParseBatch`
5. VLA 分配 `vecs[2147483647]` 触发栈溢出
6. Executor 进程崩溃 (SIGSEGV)

**效果**：
- 目标 Executor 崩溃
- Spark 任务失败重试
- 大规模攻击可导致集群 DoS

#### 场景 2: Shuffle 数据篡改

**前提条件**：
- Shuffle 数据传输未加密
- 或 External Shuffle Service 被入侵

**攻击步骤**：
1. 在 Shuffle 数据传输过程中篡改 protobuf 内容
2. 修改 `vecCnt` 字段为超大值
3. 目标 Executor 反序列化时崩溃

#### 场景 3: Spark 应用漏洞链

如果攻击者能控制输入数据并影响 Shuffle 输出格式，可以通过精心构造的数据触发整个漏洞链：
- JNI-003: ParseFromArray 失败返回无效对象
- JNI-008: VLA 栈溢出
- JNI-015: offsets 越界访问
- JNI-004: Java 数组越界写入

## 6. 缓解措施建议

### 6.1 立即修复建议

#### 修复代码 (推荐方案)

```cpp
// cpp-omni/src/jni/deserializer.cpp:169-220
JNIEXPORT void JNICALL
Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_rowShuffleParseBatch(
    JNIEnv *env, jobject obj, jlong address, jintArray typeIdArray, jintArray precisionArray,
    jintArray scaleArray, jlongArray vecNativeIdArray)
{
    spark::ProtoRowBatch* protoRowBatch = reinterpret_cast<spark::ProtoRowBatch*>(address);
    int32_t vecCount = protoRowBatch->veccnt();
    int32_t rowCount = protoRowBatch->rowcnt();
    
    // 1. vecCount 范围验证
    const int32_t MAX_VEC_COUNT = 10000;  // 合理上限
    const int32_t MAX_ROW_COUNT = 1000000;
    
    if (vecCount <= 0) {
        env->ThrowNew(runtimeExceptionClass, "Invalid vecCount: must be positive");
        return;
    }
    if (vecCount > MAX_VEC_COUNT) {
        env->ThrowNew(runtimeExceptionClass, 
            ("vecCount exceeds limit: " + std::to_string(vecCount) + " > " + std::to_string(MAX_VEC_COUNT)).c_str());
        return;
    }
    
    // 2. rowCount 范围验证
    if (rowCount <= 0 || rowCount > MAX_ROW_COUNT) {
        env->ThrowNew(runtimeExceptionClass, "Invalid rowCount");
        return;
    }
    
    // 3. vecTypes 数量与 vecCount 一致性验证
    if (protoRowBatch->vectypes_size() != vecCount) {
        env->ThrowNew(runtimeExceptionClass, 
            ("vecTypes size mismatch: " + std::to_string(protoRowBatch->vectypes_size()) + 
             " != " + std::to_string(vecCount)).c_str());
        return;
    }
    
    // 4. Java 数组长度验证
    jint typeIdArrayLen = env->GetArrayLength(typeIdArray);
    jint precisionArrayLen = env->GetArrayLength(precisionArray);
    jint scaleArrayLen = env->GetArrayLength(scaleArray);
    jint vecNativeIdArrayLen = env->GetArrayLength(vecNativeIdArray);
    
    if (typeIdArrayLen < vecCount || precisionArrayLen < vecCount ||
        scaleArrayLen < vecCount || vecNativeIdArrayLen < vecCount) {
        env->ThrowNew(runtimeExceptionClass, 
            "Java array length insufficient for vecCount");
        return;
    }
    
    // 5. 替换 VLA 为 std::vector（堆分配）
    std::vector<omniruntime::vec::BaseVector*> vecs(vecCount, nullptr);
    std::vector<omniruntime::type::DataTypeId> omniDataTypeIds(vecCount);
    
    // ... 后续代码使用 vecs[i] 代替 vecs[i]
    
    JNI_FUNC_START
    // 原有逻辑
    // ...
    
    // 修改清理逻辑
    JNI_FUNC_END_WITH_VECTOR_VEC(runtimeExceptionClass, vecs, pos)
}
```

#### 关键修复要点

1. **替换 VLA 为 std::vector** - 消除栈溢出风险
2. **添加 vecCount 上限检查** - 防止超大值
3. **添加 vecTypes.size() 一致性检查** - 防止越界访问
4. **验证 Java 数组长度** - 防止 Java 数组越界写入
5. **添加 rowCount 范围检查** - 完整输入验证

namespace SafeJniParser {
    constexpr int32_t MAX_VEC_COUNT = 10000;
    constexpr int32_t MAX_ROW_COUNT = 1000000;
    constexpr int32_t MAX_PROTOBUF_SIZE = 1024 * 1024 * 1024;  // 1GB
    
    template<typename ProtoType>
    static ProtoType* safeParse(JNIEnv* env, jlong address, jint length) {
        // 参数验证
        if (address == 0) {
            env->ThrowNew(runtimeExceptionClass, "Null address");
            return nullptr;
        }
        if (length < 0 || length > MAX_PROTOBUF_SIZE) {
            env->ThrowNew(runtimeExceptionClass, "Invalid length");
            return nullptr;
        }
        
        auto* obj = new ProtoType();
        if (!obj->ParseFromArray(reinterpret_cast<char*>(address), length)) {
            delete obj;
            env->ThrowNew(runtimeExceptionClass, "Protobuf parse failed");
            return nullptr;
        }
        
        return obj;
    }
    
    static bool validateVecCount(JNIEnv* env, int32_t vecCount) {
        if (vecCount <= 0 || vecCount > MAX_VEC_COUNT) {
            env->ThrowNew(runtimeExceptionClass, 
                ("Invalid vecCount: " + std::to_string(vecCount)).c_str());
            return false;
        }
        return true;
    }
}

## 7. 附录

### 7.1 CWE-121 定义

> **CWE-121: Stack-based Buffer Overflow**
> 
> A stack-based buffer overflow condition is a condition where the buffer being overwritten is allocated on the stack (i.e., is a local variable or, rarely, a parameter to a function).
> 
> **Consequences**:
> - DoS: Crash, Exit, or Restart
> - Execute Code: Buffer overflows often allow attackers to execute arbitrary code
> - Memory Corruption: Overwriting variables or other memory

### 7.2 VLA 安全性讨论

VLA (Variable Length Array) 是 C99 特性，但在安全敏感场景中应避免使用：

**问题**：
1. 栈空间有限，无法处理大数据
2. 运行时大小来自不可信源时风险极高
3. 无法检测分配失败（不像 malloc 返回 NULL）
4. 某些编译器不支持或行为不一致

**替代方案**：
- `std::vector` (C++) - 堆分配，可检测失败
- `alloca()` - 显式栈分配，但同样有风险
- 动态分配并检查返回值

### 7.3 相关 CVE 参考

类似漏洞：
- CVE-2023-44487: Stack overflow via unvalidated size in native code
- CVE-2022-45943: VLA stack overflow in video codec

### 7.4 修复优先级建议

| 优先级 | 建议 |
|--------|------|
| **P0 (立即)** | 替换 VLA 为 std::vector |
| **P1 (短期)** | 添加 vecCount 上限检查 |
| **P2 (中期)** | 添加 Java 数组长度验证 |
| **P3 (长期)** | 重构为安全 JNI 解析框架 |

## 8. 结论

**JNI-008 是一个真实存在的 High 级别漏洞**，需要立即修复。

**关键风险**：
- VLA 栈分配基于不可信的 protobuf 数据
- 无 vecCount 范围验证，可接受任意 int32_t 值
- 攻击者可构造恶意 Shuffle 数据触发 Executor 崩溃
- 与 JNI-007 同模式，需同步修复

**修复核心**：
1. **立即替换 VLA 为 std::vector** - 消除栈溢出风险
2. **添加 vecCount 范围验证** - 上限检查
3. **验证 vecTypes.size() 与 vecCount 一致性**
4. **验证 Java 数组长度与 vecCount 匹配**

**建议立即实施 P0 级别修复**，然后逐步完善防护措施。
