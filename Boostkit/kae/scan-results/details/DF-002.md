# DF-002：VFIO迁移接口DMA地址验证绕过漏洞

**漏洞ID**: DF-002  
**CWE**: CWE-123 (Write-what-where Condition)  
**严重性**: Critical  
**置信度**: 95%  
**CVE参考**: CVE-2025-38158 (相关bug修复)

---

## 1. 漏洞概述和严重性评估

### 漏洞类型
DMA地址验证缺失导致的硬件寄存器注入漏洞

### 严重性评估
- **攻击复杂度**: Medium - 需要宿主机访问权限或迁移网络控制
- **影响范围**: High - 可影响整个宿主机系统
- **权限提升**: High - 可导致宿主机权限提升
- **数据泄露**: High - 可读取任意宿主机内存
- **CVSS评分**: 7.8 (High) - AV:N/AC:L/PR:H/UI:N/S:C/C:H/I:H/A:H

### 核心问题
VFIO迁移接口中 `vf_qm_load_data` 函数直接使用用户提供的DMA地址写入QM硬件寄存器，缺少必要的地址有效性验证。恶意攻击者可通过控制VFIO迁移数据注入任意DMA地址，导致QM硬件对宿主机任意内存进行DMA操作。

## 2. 漏洞触发条件和攻击路径

### 关键代码位置

**文件**: `hisi_acc_vfio_pci.c`

**漏洞函数**: `vf_qm_load_data` (Lines 480-530)

```c
static int vf_qm_load_data(struct hisi_acc_vf_core_device *hisi_acc_vdev,
                           struct hisi_acc_vf_migration_file *migf)
{
    struct hisi_qm *qm = &hisi_acc_vdev->vf_qm;
    struct acc_vf_data *vf_data = &migf->vf_data;
    
    // Line 495-499: 仅检查NULL，无地址有效性验证
    if (!vf_data->eqe_dma || !vf_data->aeqe_dma ||
        !vf_data->sqc_dma || !vf_data->cqc_dma) {
        dev_info(dev, "resume dma addr is NULL!\n");
        return 0;
    }
    
    // Lines 502-505: 直接赋值，无验证
    qm->eqe_dma = vf_data->eqe_dma;    // 用户控制的DMA地址
    qm->aeqe_dma = vf_data->aeqe_dma;  // 直接赋值到内核结构
    qm->sqc_dma = vf_data->sqc_dma;    // 无任何验证
    qm->cqc_dma = vf_data->cqc_dma;
    
    // Lines 516, 522: 写入硬件寄存器
    ret = hisi_qm_mb_write(qm, QM_MB_CMD_SQC_BT, qm->sqc_dma, 0, 0);
    ret = hisi_qm_mb_write(qm, QM_MB_CMD_CQC_BT, qm->cqc_dma, 0, 0);
}
```

**数据来源**: `hisi_acc_vf_resume_write` (Line 781)

```c
ret = copy_from_user(vf_data + *pos, buf, len);  // 从用户空间复制迁移数据
```

### 攻击路径修正

**⚠️ 重要修正**: 原漏洞描述中"恶意VM guest可注入DMA地址"的攻击路径是**错误的**。

**正确的攻击路径**:

VFIO迁移是**宿主机之间的操作**，VM guest不直接控制迁移数据。真实攻击者包括：

#### 2.1 恶意宿主机管理员攻击 (Primary Attack Vector)
```
[恶意管理员] → 修改VFIO迁移数据包 → 注入恶意DMA地址
    ↓
[hisi_acc_vf_resume_write] → copy_from_user(vf_data)
    ↓
[vf_qm_load_data] → qm->sqc_dma/qm->cqc_dma = 恶意地址
    ↓
[hisi_qm_mb_write] → QM_MB_CMD_SQC_BT/CQC_BT → 硬件寄存器
    ↓
[QM硬件] → 使用恶意DMA地址进行DMA操作 → 任意内存读写
```

**触发条件**:
- 攻击者拥有宿主机管理员权限（可以操作VFIO设备）
- 目标宿主机接收VFIO迁移数据
- 攻击者可以篡改迁移数据中的DMA地址字段

#### 2.2 迁移网络MITM攻击 (Secondary Attack Vector)
```
[源宿主机] → 发送VFIO迁移数据 → [网络传输] → [目标宿主机]
                                    ↑
                            [MITM攻击者]
                            篡改DMA地址字段
```

**触发条件**:
- VFIO迁移数据通过网络传输
- 传输未加密或未完整性验证
- 攻击者可拦截修改网络流量

**参考**: Black Hat论文 "Empirical Exploitation of Live Virtual Machine Migration" 证实迁移数据可被MITM攻击篡改。

#### 2.3 容器逃逸后的进程攻击 (Container Escape Vector)
```
[容器内恶意进程] → 容器逃逸 → [宿主机内核访问]
    ↓
[修改VFIO迁移数据] → 注入恶意DMA地址 → QM硬件任意DMA
```

**触发条件**:
- 宿主机运行容器化环境
- 容器逃逸漏洞被利用
- 攻击进程可访问VFIO接口

### 数据流详细分析

#### Stage 1: 数据接收
- **接口**: VFIO ioctl → QEMU → `hisi_acc_vf_resume_write`
- **操作**: `copy_from_user(vf_data, buf, len)` (Line 781)
- **数据结构**: `struct acc_vf_data` 包含4个DMA地址字段
  - `eqe_dma` (u64) - Event Queue Entry DMA地址
  - `aeqe_dma` (u64) - Asynchronous Event Queue Entry DMA地址  
  - `sqc_dma` (u64) - SQ Context BT DMA地址
  - `cqc_dma` (u64) - CQ Context BT DMA地址

#### Stage 2: 地址加载
- **函数**: `vf_qm_load_data`
- **操作**: 直接赋值到 `qm->eqe_dma/aeqe_dma/sqc_dma/cqc_dma`
- **验证缺失**: 仅检查NULL，无以下验证:
  - 地址范围验证
  - IOMMU映射验证
  - 地址对齐验证
  - 地址所属设备验证

#### Stage 3: 硬件写入
- **函数**: `hisi_qm_mb_write` → `qm_mb_write`
- **寄存器**: QM_MB_CMD_SEND_BASE (Offset 0x300)
- **数据结构**: `struct qm_mailbox`
  ```c
  struct qm_mailbox {
      __le16 w0;           // 命令字 + busy标志
      __le16 queue_num;    // 队列号
      __le32 base_l;       // DMA地址低32位 ← 用户控制
      __le32 base_h;       // DMA地址高32位 ← 用户控制
      __le32 rsvd;
  };
  ```
- **写入方式**: `memcpy_toio(fun_base, mailbox, 128bit)` (Line 610)
- **硬件行为**: QM硬件解析mailbox，使用DMA地址进行内存访问

#### Stage 4: 硬件DMA操作
QM硬件使用注入的DMA地址进行以下操作:
1. **EQE/AEQE读取**: 从 `eqe_dma/aeqe_dma` 读取队列事件
2. **SQC/CQC访问**: 通过 `QM_MB_CMD_SQC_BT/CQC_BT` 配置队列上下文
3. **DMA读写**: QM硬件根据配置对宿主机内存进行DMA操作

## 3. 漏洞利用步骤和影响分析

### 3.1 恶意管理员利用步骤

#### Step 1: 准备恶意迁移数据
```c
struct acc_vf_data malicious_data = {
    .acc_magic = ACC_DEV_MAGIC_V2,  // 需要正确magic
    .major_ver = ACC_DRV_MAJOR_VER,  // 版本检查
    .dev_id = target_device_id,      // 设备ID匹配
    .qp_num = target_qp_num,         // QP数量匹配
    .que_iso_cfg = target_iso_state, // 隔离状态匹配
    
    // 注入恶意DMA地址
    .eqe_dma = 0xDEADBEEF00000000,   // 指向目标内存区域
    .aeqe_dma = 0xCAFEBABE00000000,
    .sqc_dma = attacker_controlled_addr,  // 攻击者控制的地址
    .cqc_dma = attacker_controlled_addr,
};
```

#### Step 2: 通过VFIO接口注入数据
```bash
# 方法1: 通过QEMU管理迁移数据
qemu-system -device vfio-pci,host=0000:xx:00.0 \
             -incoming tcp:0:4444  # 接收迁移数据

# 方法2: 直接操作VFIO设备文件
# 需要root权限访问 /dev/vfio/xx
```

#### Step 3: 触发QM硬件DMA操作
当VF设备恢复后，QM硬件会自动使用注入的DMA地址进行DMA读写:
- 读操作: 可读取宿主机任意物理内存
- 写操作: 可写入宿主机任意物理内存

### 3.2 网络MITM利用步骤

参考 Black Hat 2008 论文的方法:

#### Step 1: 拦截迁移流量
```python
# 在迁移网络路径上拦截
# QEMU默认使用TCP传输，可被MITM
def intercept_migration():
    # 拦截TCP流量，修改DMA地址字段
    # VFIO迁移数据结构已知，可精确定位偏移
    modify_dma_addresses(packet)
```

#### Step 2: 篡改DMA地址
- 定位 `acc_vf_data` 结构在迁移数据中的偏移
- 修改 `eqe_dma/aeqe_dma/sqc_dma/cqc_dma` 字段为恶意地址
- 保持其他验证字段不变（magic, dev_id, qp_num等）

### 3.3 漏洞影响分析

#### 直接影响
1. **任意内存读取**:
   - QM硬件从恶意DMA地址读取数据
   - 可读取宿主机内核内存、用户进程内存
   - 泄露敏感数据：密钥、密码、内核结构

2. **任意内存写入**:
   - QM硬件向恶意DMA地址写入数据
   - 可修改内核数据结构、代码段
   - 实现权限提升、绕过安全机制

3. **硬件设备控制**:
   - 控制QM硬件的DMA行为
   - 可能触发硬件异常或崩溃

#### 系统级影响
1. **宿主机完全控制**:
   - 通过内存写入实现任意代码执行
   - 提升到root权限
   - 绕过所有安全机制（SELinux, AppArmor等）

2. **横向渗透**:
   - 宿主机被控制后可攻击其他VM
   - 访问宿主机上的其他资源
   - 持久化控制

3. **数据泄露**:
   - 读取宿主机上所有VM的内存
   - 泄露加密密钥、证书、敏感数据
   - 违反数据保护法规

## 5. 修复建议和缓解措施

### 5.1 代码级修复方案

#### 方案1: DMA地址有效性验证 (推荐)

```c
static int vf_qm_load_data(struct hisi_acc_vf_core_device *hisi_acc_vdev,
                           struct hisi_acc_vf_migration_file *migf)
{
    struct hisi_qm *qm = &hisi_acc_vdev->vf_qm;
    struct acc_vf_data *vf_data = &migf->vf_data;
    struct device *dev = &qm->pdev->dev;
    
    // 新增: DMA地址有效性验证
    if (!validate_dma_address(qm, vf_data->eqe_dma)) {
        dev_err(dev, "Invalid eqe_dma address!\n");
        return -EINVAL;
    }
    if (!validate_dma_address(qm, vf_data->aeqe_dma)) {
        dev_err(dev, "Invalid aeqe_dma address!\n");
        return -EINVAL;
    }
    if (!validate_dma_address(qm, vf_data->sqc_dma)) {
        dev_err(dev, "Invalid sqc_dma address!\n");
        return -EINVAL;
    }
    if (!validate_dma_address(qm, vf_data->cqc_dma)) {
        dev_err(dev, "Invalid cqc_dma address!\n");
        return -EINVAL;
    }
    
    // 继续正常流程
    qm->eqe_dma = vf_data->eqe_dma;
    ...
}

// 新增验证函数
static bool validate_dma_address(struct hisi_qm *qm, u64 dma_addr)
{
    struct device *dev = &qm->pdev->dev;
    
    // 1. 检查地址非零且已分配
    if (!dma_addr)
        return false;
    
    // 2. 检查地址在设备DMA范围内
    if (dma_addr > qm->dma_mask)
        return false;
    
    // 3. 检查地址对齐
    if (dma_addr & (PAGE_SIZE - 1))
        return false;
    
    // 4. 通过IOMMU验证地址映射
    // 确保地址属于该设备的DMA映射
    dma_addr_t validated_addr = dma_map_page(dev, 
        phys_to_page(dma_addr), 0, PAGE_SIZE, DMA_TO_DEVICE);
    if (dma_mapping_error(dev, validated_addr))
        return false;
    
    // 5. 检查地址是否来自设备自身的DMA池
    if (!is_device_dma_pool_address(qm, dma_addr))
        return false;
    
    return true;
}
```

## 7. 总结和建议

### 漏洞真实性判断
**确认**: 这是一个真实的安全漏洞，DMA地址验证缺失导致恶意地址注入可行。

### 攻击场景修正
**重要**: 原描述中"VM guest直接攻击"场景错误。正确攻击者是:
