# JNI-006：VLA栈溢出漏洞

## 漏洞概述

**漏洞类型**: 栈缓冲区溢出 (CWE-121)  
**严重级别**: High  
**置信度**: 85%  
**影响模块**: jni_bindings

OmniOperator 的 JNI 函数 `createSortOperatorFactory` 使用变长数组 (VLA)，数组大小直接来自 JNI 输入的数组长度，未设置上限。恶意 Java 代码可传入极大数组导致栈溢出。

## 漏洞触发条件

1. 攻击者能够在 JVM 中运行 Java 代码
2. 攻击者能够调用 OmniSortOperatorFactory 的 JNI 接口
3. 攻击者能够传入极大长度的数组

## 关键代码分析

### jni_operator_factory.cpp:288-318

```cpp
JNIEXPORT jlong JNICALL Java_..._createSortOperatorFactory(
    JNIEnv *env, jclass jObj, jstring jSourceTypes, jintArray jOutputCols,
    jobjectArray jSortCols, jintArray jAscendings, jintArray jNullFirsts, ...)
{
    auto sourceTypesCharPtr = env->GetStringUTFChars(jSourceTypes, JNI_FALSE);
    jint *outputColsArr = env->GetIntArrayElements(jOutputCols, JNI_FALSE);
    auto outputColsCount = env->GetArrayLength(jOutputCols);
    
    // ⚠️ 关键缺陷: sortColsCount 来自 JNI 输入，无上限验证
    auto sortColsCount = env->GetArrayLength(jSortCols);
    
    // ⚠️ VLA 在栈上分配，大小完全由外部控制
    int32_t sortColsArr[sortColsCount];  // 如果 sortColsCount 很大 → 栈溢出
    
    GetColumnsFromExpressions(env, jSortCols, sortColsArr, sortColsCount);
    ...
}
```

**安全缺陷分析**:
- `GetArrayLength` 返回的值直接用于 VLA 大小
- VLA 在栈上分配内存，默认栈大小通常只有几 MB
- 传入极大数组长度会导致栈溢出
- 没有最大长度限制检查

### 其他 VLA 使用

## 利用步骤 (PoC)

### Java 攻击代码

```java
// StackOverflowAttack.java
public class StackOverflowAttack {
    // JNI 接口
    private native long createSortOperatorFactory(
        String sourceTypes,
        int[] outputCols,
        int[] sortCols,  // ⚠️ 可控大小
        int[] ascendings,
        int[] nullFirsts,
        String operatorConfig
    );
    
    public void attack() {
        // 攻击方式 1: 直接栈溢出
        int[] hugeArray = new int[100000000];  // 100 million elements
        createSortOperatorFactory("...", ..., hugeArray, ...);
        // → Native 栈溢出 → Crash
        
        // 攻击方式 2: 精心计算栈大小
        // 假设栈大小 8MB，int32_t 占 4 bytes
        // 8MB / 4 = 2000000 elements 可以触发栈边界
        int[] calculatedArray = new int[2500000];
        createSortOperatorFactory("...", ..., calculatedArray, ...);
        // → 可能部分成功，但覆盖其他栈数据
    }
}
```

### 攻击效果分析

| 数组大小 | 栈分配 | 效果 |
|---------|--------|------|
| 1,000 | 4KB | 正常运行 |
| 100,000 | 400KB | 可能正常 |
| 1,000,000 | 4MB | 可能触发栈边界 |
| 10,000,000 | 40MB | 栈溢出 → Crash |
| 100,000,000 | 400MB | 明确栈溢出 |

## 危害评估

### 攻击影响
- **Denial of Service**: 使 Worker 进程崩溃
- **内存破坏**: 栈溢出可能覆盖函数返回地址或其他重要数据
- **潜在的代码执行**: 如果精心构造，可能覆盖返回地址实现代码执行（难度较高）

### CVSS 评分预估
- **攻击向量**: Local
- **攻击复杂度**: Low (DoS) / High (代码执行)
- **权限要求**: Low
- **影响**: High (DoS) / Critical (代码执行)

**预估 CVSS 评分**: 7.5 (High)

## 修复建议

### 1. 添加上限检查（优先级：高）

```cpp
JNIEXPORT jlong JNICALL Java_..._createSortOperatorFactory(...)
{
    auto sortColsCount = env->GetArrayLength(jSortCols);
    
    // 添加最大长度限制
    const int32_t MAX_SORT_COLS = 1024;  // 设置合理的上限
    if (sortColsCount > MAX_SORT_COLS) {
        env->ThrowNew(omniRuntimeExceptionClass, 
            "Sort columns count exceeds maximum limit");
        return 0;
    }
    
    // 现在可以安全使用 VLA
    int32_t sortColsArr[sortColsCount];
    ...
}
```

### 2. 使用动态内存分配替代 VLA（优先级：高）

```cpp
JNIEXPORT jlong JNICALL Java_..._createSortOperatorFactory(...)
{
    auto sortColsCount = env->GetArrayLength(jSortCols);
    
    // 添加上限检查
    if (sortColsCount > MAX_SORT_COLS) {
        ...
    }
    
    // 使用堆内存替代 VLA
    std::vector<int32_t> sortColsArr(sortColsCount);
    // 或
    std::unique_ptr<int32_t[]> sortColsArr(new int32_t[sortColsCount]);
    
    GetColumnsFromExpressions(env, jSortCols, sortColsArr.data(), sortColsCount);
    ...
}
```

### 3. 修复所有 VLA 使用点（优先级：中）

```cpp
// Transform 函数同样需要修复

```
