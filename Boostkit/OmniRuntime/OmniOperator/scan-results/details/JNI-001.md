# JNI-001：JNI空指针解引用漏洞

## 漏洞概述

**漏洞类型**: 空指针解引用 (CWE-476)  
**严重级别**: High (原 Critical)  
**置信度**: 85%  
**影响模块**: jni_bindings, operator

OmniOperator 的 JNI 入口函数 `addInputNative` 直接将 Java 端传入的 `jlong` 地址转换为 C++ 指针使用，未进行有效性验证。恶意 Java 代码可传入无效地址导致 Native 侧空指针解引用或任意地址访问。

## 漏洞触发条件

1. 攻击者能够在 JVM 中运行 Java 代码
2. 攻击者能够调用 OmniOperator 的 JNI 接口
3. 攻击者能够传入无效或恶意构造的地址值

## 关键代码分析

### jni_operator.cpp:159-170

```cpp
JNIEXPORT jint JNICALL Java_nova_hetu_omniruntime_operator_OmniOperator_addInputNative(
    JNIEnv *env, jobject jObj,
    jlong jOperatorAddress, jlong jVecBatchAddress)
{
    int32_t errNo = 0;
    // ⚠️ 无空指针检查，直接 reinterpret_cast
    auto *vecBatch = reinterpret_cast<VectorBatch *>(jVecBatchAddress);
    auto *nativeOperator = reinterpret_cast<op::Operator *>(jOperatorAddress);
    
    JNI_METHOD_START  // 有异常处理，但不防止空指针解引用
    RecordInputVectorsStack(vecBatch, env);  // ⚠️ vecBatch 可能为 nullptr
    nativeOperator->SetInputVecBatch(vecBatch);  // ⚠️ 解引用
    errNo = nativeOperator->AddInput(vecBatch);  // ⚠️ 解引用
    JNI_METHOD_END_WITH_VECBATCH(errNo, nativeOperator->GetInputVecBatch())
    return errNo;
}
```

**安全缺陷分析**:
- `reinterpret_cast` 将 `jlong` 转换为指针，无任何验证
- 转换后的指针立即被解引用调用方法
- `JNI_METHOD_START` 只提供异常处理，不检查指针有效性
- 如果传入 0 或无效地址，会导致空指针解引用或访问违规

## 利用步骤 (PoC)

### 恶意 Java 代码攻击

```java
// MaliciousAdapter.java
public class MaliciousAdapter {
    // 获取 OmniOperator 的 JNI 接口
    private native int addInputNative(long operatorAddress, long vecBatchAddress);
    
    public void crashNative() {
        // 攻击方式 1: 传入空地址导致空指针解引用
        addInputNative(0, 0);  // → Native crash
        
        // 攻击方式 2: 传入无效地址导致访问违规
        addInputNative(0xdeadbeef, 0xcafebabe);  // → SEGFAULT
        
        // 攻击方式 3: 传入部分有效地址，部分无效
        // 假设我们有一个有效的 operator 地址
        long validOperator = getValidOperator();
        addInputNative(validOperator, 0);  // → vecBatch 空指针解引用
    }
}
```

### 攻击效果

| 输入地址 | Native 行为 | 效果 |
|---------|-------------|------|
| `0` | 空指针解引用 | Crash / SIGSEGV |
| `0xdeadbeef` | 无效地址访问 | Crash / SIGSEGV |
| 部分有效 | 部分解引用成功 | 内存破坏 / 未定义行为 |

## 危害评估

### 攻击影响
- **Denial of Service**: 使 Spark/Hive Worker 进程崩溃
- **内存破坏**: 如果传入精心构造的地址，可能导致内存破坏
- **信息泄露**: 异常处理可能泄露内存布局信息

### 攻击条件分析
- **信任边界**: 根据项目模型，JNI 接口是 "semi_trusted"，调用方是同一 JVM 进程
- **攻击者前提**: 需要能够在 JVM 中运行代码
- **攻击价值**: 如果攻击者已在 JVM 中，可直接执行更多操作

**评估**: 此漏洞在攻击者已获得 JVM 代码执行能力时可被利用。主要危害是 DoS 和可能的内存破坏。

- **权限要求**: Low
- **影响**: High (进程崩溃)

## 修复建议

### 1. 添加空指针检查（优先级：高）

```cpp
JNIEXPORT jint JNICALL Java_..._addInputNative(...)
{
    int32_t errNo = 0;
    auto *vecBatch = reinterpret_cast<VectorBatch *>(jVecBatchAddress);
    auto *nativeOperator = reinterpret_cast<op::Operator *>(jOperatorAddress);
    
    // 添加空指针检查
    if (vecBatch == nullptr || nativeOperator == nullptr) {
        env->ThrowNew(omniRuntimeExceptionClass, "Invalid null pointer passed to JNI");
        return -1;
    }
    
    // 可选: 添加地址有效性检查
    if (!IsValidPointer(vecBatch) || !IsValidPointer(nativeOperator)) {
        env->ThrowNew(omniRuntimeExceptionClass, "Invalid pointer address");
        return -1;
    }
    
    JNI_METHOD_START
    ...
}
```

### 2. 使用安全转换函数（优先级：中）

```cpp
// 创建安全指针转换辅助函数
template<typename T>
T* SafePtrFromJlong(JNIEnv* env, jlong addr) {
    if (addr == 0) {
        env->ThrowNew(omniRuntimeExceptionClass, "Null pointer");
        return nullptr;
    }
    return reinterpret_cast<T*>(addr);
}
```

### 3. 添加指针跟踪机制（优先级：低）

```cpp
// 维护有效指针的跟踪列表，验证传入地址是否在有效范围
static std::set<void*> validOperators;
static std::set<void*> validVecBatches;
```
