# JNI-009: Memory Leak - Missing ReleaseStringUTFChars

## Vulnerability Summary

| Attribute | Value |
|-----------|-------|
| **Vulnerability ID** | JNI-009 |
| **Type** | Memory Leak (CWE-401) |
| **Severity** | Medium |
| **Confidence** | 85% |
| **File** | `cpp-omni/src/jni/SparkJniWrapper.cpp` |
| **Lines** | 53-67 |
| **Function** | `Java_com_huawei_boostkit_spark_jni_SparkJniWrapper_nativeMake` |

## Description

The function `Java_com_huawei_boostkit_spark_jni_SparkJniWrapper_nativeMake` acquires a JNI string resource via `GetStringUTFChars` for the `jInputType` parameter but never releases it with `ReleaseStringUTFChars`. This causes a persistent memory leak in JNI string resources.

### Vulnerable Code

```cpp
// Line 53: String resource acquired
const char *inputTypeCharPtr = env->GetStringUTFChars(jInputType, JNI_FALSE);
DataTypes inputVecTypes = Deserialize(inputTypeCharPtr);
const int32_t *inputVecTypeIds = inputVecTypes.GetIds();
//
std::vector<DataTypePtr> inputDataTypes = inputVecTypes.Get();
int32_t size = inputDataTypes.size();
uint32_t *inputDataPrecisions = new uint32_t[size];
uint32_t *inputDataScales = new uint32_t[size];
for (int i = 0; i < size; ++i) {
    if (inputDataTypes[i]->GetId() == OMNI_DECIMAL64 || inputDataTypes[i]->GetId() == OMNI_DECIMAL128) {
        inputDataScales[i] = std::dynamic_pointer_cast<DecimalDataType>(inputDataTypes[i])->GetScale();
        inputDataPrecisions[i] = std::dynamic_pointer_cast<DecimalDataType>(inputDataTypes[i])->GetPrecision();
    }
}
inputDataTypes.clear();
// Line 67: Missing env->ReleaseStringUTFChars(jInputType, inputTypeCharPtr);
```

**Missing release call:**
```cpp
env->ReleaseStringUTFChars(jInputType, inputTypeCharPtr);  // NEVER CALLED
```

### Alternative Safe Pattern Exists

The codebase has a safer utility function `JStringToCString` that avoids the release problem entirely:

```cpp
// jni_common.h lines 69-76
static inline std::string JStringToCString(JNIEnv *env, jstring string)
{
    int32_t clen = env->GetStringUTFLength(string);
    int32_t jlen = env->GetStringLength(string);
    char buffer[clen];
    env->GetStringUTFRegion(string, 0, jlen, buffer);  // Copies to buffer, no release needed
    return std::string(buffer, clen);
}
```

This pattern is used correctly at line 288:
```cpp
auto spillDirStr = JStringToCString(env, spillDir);  // Safe - no release needed
```

## Root Cause Analysis

1. **Inconsistent resource management**: The developer properly releases other string resources but forgot to release `inputTypeCharPtr`.

2. **No RAII wrapper**: JNI resources are manually managed without RAII guards, making leaks easy to introduce.

3. **Exception safety gap**: Even if `ReleaseStringUTFChars` were added at the end, the `JNI_FUNC_START`/`JNI_FUNC_END` macros only catch exceptions after the fact - they don't guarantee resource cleanup during stack unwinding.

## Impact Assessment

### Technical Impact

| Impact Area | Severity |
|-------------|----------|

| **Service Degradation** | Medium |
| **Crash Risk** | Medium |

### Business Impact

1. **Long-running Spark Executors**: Spark executors typically run for extended periods processing many shuffle operations. Each call to `nativeMake` leaks JNI string resources.

2. **Accumulative Effect**: The leak compounds over time. A busy executor processing thousands of shuffle operations will gradually consume more memory.

3. **Production Impact**: 
   - Gradual memory exhaustion leading to OOM errors
   - Increased GC pressure in the JVM
   - Potential executor crashes requiring task re-execution
   - Reduced cluster efficiency

### Attack Scenario

An attacker could exploit this by:
1. Submitting jobs with many small shuffle operations
2. Each operation triggers the leak
3. Executors gradually run out of native memory
4. Eventually causing service disruption

## Recommended Fix

### Option 1: Add Release Call (Minimal Change)

```cpp
const char *inputTypeCharPtr = env->GetStringUTFChars(jInputType, JNI_FALSE);
DataTypes inputVecTypes = Deserialize(inputTypeCharPtr);
// ... use inputTypeCharPtr ...
env->ReleaseStringUTFChars(jInputType, inputTypeCharPtr);  // ADD THIS LINE
```

**Placement**: The release should be added after line 66, before `inputDataTypes.clear()`.

### Option 2: Use JStringToCString (Preferred)

```cpp
// Replace lines 53-54 with:
std::string inputTypeStr = JStringToCString(env, jInputType);
DataTypes inputVecTypes = Deserialize(inputTypeStr.c_str());
// No release needed - JStringToCString handles it internally
```

**Benefits**:
- Consistent with the pattern used at line 288
- Exception-safe (uses stack-allocated buffer)
- No manual resource management needed

### Option 3: RAII Guard (Best Practice)

Create a RAII wrapper for JNI string resources:

```cpp
// jni_common.h
class JniStringGuard {
public:
    JniStringGuard(JNIEnv* env, jstring str) 
        : env_(env), str_(str), chars_(env->GetStringUTFChars(str, nullptr)) {}
    
    ~JniStringGuard() {
        if (chars_) env_->ReleaseStringUTFChars(str_, chars_);
    }
    
    const char* get() const { return chars_; }
    operator const char*() const { return chars_; }
    
    // Non-copyable, movable
    JniStringGuard(const JniStringGuard&) = delete;
    JniStringGuard& operator=(const JniStringGuard&) = delete;
    JniStringGuard(JniStringGuard&& other) noexcept 
        : env_(other.env_), str_(other.str_), chars_(other.chars_) {
        other.chars_ = nullptr;
    }
    
private:
    JNIEnv* env_;
    jstring str_;
    const char* chars_;
};

// Usage:
JniStringGuard inputTypeGuard(env, jInputType);
DataTypes inputVecTypes = Deserialize(inputTypeGuard.get());
// Automatic release when guard goes out of scope
```
