# JNI-013 漏洞详细分析报告

## 1. 漏洞概述

### 基本信息
| 属性 | 值 |
|------|-----|
| **漏洞 ID** | JNI-013 |
| **类型** | Use After Free (CWE-416) |
| **严重性** | High |
| **置信度** | 85/100 |
| **状态** | CONFIRMED |
| **文件路径** | `cpp-omni/src/jni/SparkJniWrapper.cpp` |
| **行号** | 218-229 |
| **函数名** | `Java_com_huawei_boostkit_spark_jni_SparkJniWrapper_close` |
| **信任边界** | JNI Interface (High Risk) - Spark JVM Process → Native C++ Library |

### 漏洞摘要
JNI close 函数删除 splitter 指针后未将指针置为 NULL。如果同一 handle 被多次调用 close，将导致 Use After Free 漏洞：第一次调用释放内存，后续调用尝试再次 delete 已释放的内存地址，触发 double-free 或访问已释放的内存。

### 关联漏洞
该漏洞与以下位置存在相同模式：
- **SparkJniWrapper.cpp:426** (`nativeClose`): 相同的 delete 后未置 NULL 模式
- **deserializer.cpp:44,145,212**: delete 后未置 NULL 的类似模式

## 2. 技术细节分析

### 2.1 漏洞代码

```cpp
// cpp-omni/src/jni/SparkJniWrapper.cpp:218-229
JNIEXPORT void JNICALL Java_com_huawei_boostkit_spark_jni_SparkJniWrapper_close(JNIEnv *env, jobject,
    jlong splitter_addr)
{
    JNI_FUNC_START
        auto splitter = reinterpret_cast<Splitter*>(splitter_addr);
        if (!splitter) {
            std::string error_message = "Invalid splitter id " + std::to_string(splitter_addr);
            env->ThrowNew(runtimeExceptionClass, error_message.c_str());
        }
        delete splitter;  // ← 漏洞核心：delete 后未置 NULL
    JNI_FUNC_END_VOID(runtimeExceptionClass)
}
```

### 2.2 漏洞根因分析

#### 问题代码逻辑

```
第一次 close(splitter_addr):
    splitter = reinterpret_cast<Splitter*>(splitter_addr)  // 获取地址
    if (!splitter) → FALSE (地址值不为0)                   // 检查通过
    delete splitter                                         // 释放内存
    // splitter_addr 值仍为原地址，未修改                   // ← 问题根源

第二次 close(splitter_addr):
    splitter = reinterpret_cast<Splitter*>(splitter_addr)  // 获取相同地址值
    if (!splitter) → FALSE (地址值仍不为0)                 // ← NULL 检查失效
    delete splitter                                         // ← UAF/Double-Free！
```

#### 关键缺陷点

1. **delete 后未置 NULL**：
   ```cpp
   delete splitter;
   // 缺失: splitter = nullptr;
   ```

2. **NULL 检查使用的是地址值而非指针状态**：
   ```cpp
   // splitter_addr 是 jlong 类型，存储的是地址数值
   // 该数值在第一次 delete 后不会自动变为 0
   // reinterpret_cast<Splitter*>(splitter_addr) 每次都返回相同的指针值
   ```

3. **无 handle 状态管理机制**：
   - 没有记录哪些 handle 已经被释放
   - 没有防止同一 handle 多次使用的机制

### 2.3 内存生命周期分析

```
┌────────────────────────────────────────────────────────────────────┐
│                    Splitter 内存生命周期                            │
└────────────────────────────────────────────────────────────────────┘

[nativeMake]
    │
    │ Splitter::Make() → new Splitter(...)
    │ splitter_addr = reinterpret_cast<jlong>(splitter)
    │ return splitter_addr  →  例如 0x7F1234567890
    ▼
[split / rowSplit / stop]
    │ 使用 splitter_addr 访问内存
    │ 正常操作...
    ▼
[close 第一次调用]
    │ splitter = reinterpret_cast<Splitter*>(0x7F1234567890)
    │ delete splitter  →  释放内存 0x7F1234567890
    │ 内存被标记为 freed，但地址值未变
    │ splitter_addr = 0x7F1234567890 (不变)
    ▼
[close 第二次调用] ← UAF 触发点
    │ splitter = reinterpret_cast<Splitter*>(0x7F1234567890)
    │ if (!splitter) → FALSE (地址值 ≠ 0)
    │ delete splitter  →  Double-Free / UAF
    │     ↓
    │  ┌─────────────────────────────────┐
    │  │ glibc 行为：                     │
    │  │ - 快速失败：SIGABRT (double-free)│
    │  │ - 或内存损坏：heap corruption   │
    │  │ - 或静默错误：可能导致后续崩溃  │
    │  └─────────────────────────────────┘
```

### 2.4 C++ delete 后置 NULL 的必要性

#### C++ 标准行为

```cpp
int* ptr = new int(42);
delete ptr;    // 释放内存
// ptr 的值不变，仍指向已释放的内存地址

// 安全做法：
delete ptr;
ptr = nullptr;  // 显式置空，防止后续误用
```

#### delete nullptr 是安全的

```cpp
int* ptr = nullptr;
delete ptr;  // C++ 标准规定：delete nullptr 无操作，安全
```

因此，delete 后置 NULL 是防御性编程的标准做法：
1. 防止后续 delete 同一指针（double-free）
2. 使 NULL 检查生效
3. 防止后续访问已释放内存

### 3.2 攻击场景分析

#### 场景 1: 线程竞争攻击

```java
// 攻击者构造多线程环境
long handle = wrapper.make(...);

// 线程 A 和 B 同时调用 close
Thread A: wrapper.close(handle);  // 第一次 delete 成功
Thread B: wrapper.close(handle);  // 第二次 delete → UAF
```

**Java 层的 stopping 标志无法防止此场景**（如果两个线程同时进入 stop 方法）。

#### 场景 2: 异常处理绕过

```java
// OmniColumnarShuffleWriter.scala
override def stop(success: Boolean): Option[MapStatus] = {
    try {
      if (stopping) {
        None
      } else {
        stopping = true  // ← 如果此处抛异常，stopping 可能未设置
        // ...
      }
    } finally {
      if (nativeSplitter != 0) {
        jniWrapper.close(nativeSplitter)  // 第一次调用
        nativeSplitter = 0
      }
    }
}

// 如果 finally 块执行前发生异常，
// nativeSplitter 可能未置 0，后续可能再次调用 close
```

#### 场景 3: 反射攻击

```java
// 攻击者通过反射直接调用 native 方法
public class Attack {
    public static void triggerUAF(long handle) {
        // 反射获取 nativeMake 创建的 handle
        long stolenHandle = ...;
        
        // 直接调用 native close（绕过 Java 层 stopping 检查）
        Method closeMethod = SparkJniWrapper.class.getDeclaredMethod("close", long.class);
        closeMethod.setAccessible(true);
        
        closeMethod.invoke(wrapper, stolenHandle);  // 第一次
        closeMethod.invoke(wrapper, stolenHandle);  // 第二次 → UAF
    }
}
```

#### 场景 4: 内存重用攻击（更危险）

```
第一次 close(handle):
    delete splitter → 内存被释放，地址 0x7F1234567890 被标记为 freed

其他操作分配内存:
    新对象被分配到同一地址 0x7F1234567890（glibc 可能重用）

第二次 close(handle):
    delete splitter → 释放新对象的内存！
    → 新对象被错误释放
    → 逻辑错误或后续崩溃

如果攻击者能控制新分配对象的内容:
    → 可能实现更复杂的攻击（如对象类型混淆）
```

### 3.3 Java 层防护分析

查看 Java 层代码：

```scala
// OmniColumnarShuffleWriter.scala:179-197
override def stop(success: Boolean): Option[MapStatus] = {
    try {
      if (stopping) {
        None
      } else {
        stopping = true
        if (success) {
          Option(mapStatus)
        } else {
          None
        }
      }
    } finally {
      if (nativeSplitter != 0) {
        jniWrapper.close(nativeSplitter)
        nativeSplitter = 0  // ← Java 层有置 0 操作
      }
    }
}
```

**Java 层防护措施**：
1. `stopping` 标志防止同一对象多次 stop
2. `nativeSplitter = 0` 置零防止后续误用

**但防护不完整的原因**：
1. **只依赖 Java 层防护**：native 层应独立防御
2. **反射可绕过**：攻击者可直接调用 native close
3. **线程竞争风险**：多线程并发时 stopping 检查可能失效
4. **异常处理不完整**：某些异常情况下 nativeSplitter 可能未置 0

#### 修复方案 C: Handle 注册表机制（最佳）

```cpp
// 新增 handle_manager.h
#include <unordered_set>
#include <mutex>

namespace HandleManager {
    static std::unordered_set<jlong> active_handles;
    static std::mutex handle_mutex;
    
    bool registerHandle(jlong handle) {
        std::lock_guard<std::mutex> lock(handle_mutex);
        if (handle == 0) return false;
        active_handles.insert(handle);
        return true;
    }
    
    bool unregisterHandle(jlong handle) {
        std::lock_guard<std::mutex> lock(handle_mutex);
        if (handle == 0) return true;  // 已经无效
        auto it = active_handles.find(handle);
        if (it == active_handles.end()) {
            return false;  // handle 不在注册表中，可能已被释放
        }
        active_handles.erase(it);
        return true;
    }
    
    bool isValidHandle(jlong handle) {
        std::lock_guard<std::mutex> lock(handle_mutex);
        return active_handles.find(handle) != active_handles.end();
    }
}

// 修复后的 close 函数
JNIEXPORT void JNICALL Java_com_huawei_boostkit_spark_jni_SparkJniWrapper_close(JNIEnv *env, jobject,
    jlong splitter_addr)
{
    JNI_FUNC_START
        // 检查 handle 是否有效
        if (!HandleManager::isValidHandle(splitter_addr)) {
            std::string error_message = "Invalid or already released splitter id " + 
                                        std::to_string(splitter_addr);
            env->ThrowNew(runtimeExceptionClass, error_message.c_str());
            return;  // ← 关键：不执行 delete
        }
        
        auto splitter = reinterpret_cast<Splitter*>(splitter_addr);
        
        // 先从注册表移除，防止并发问题
        HandleManager::unregisterHandle(splitter_addr);
        
        delete splitter;
        splitter = nullptr;
    JNI_FUNC_END_VOID(runtimeExceptionClass)
}

// nativeMake 也需要注册
JNIEXPORT jlong JNICALL Java_com_huawei_boostkit_spark_jni_SparkJniWrapper_nativeMake(...) {
    JNI_FUNC_START
        // ... 创建 splitter ...
        auto splitter = Splitter::Make(...);
        jlong handle = reinterpret_cast<jlong>(splitter);
        HandleManager::registerHandle(handle);  // ← 注册 handle
        return handle;
    JNI_FUNC_END(runtimeExceptionClass)
}
```

#### 修复方案 D: 使用智能指针（架构改进）

```cpp
// 使用 shared_ptr 管理 handle
#include <memory>
#include <unordered_map>

namespace HandleManager {
    static std::unordered_map<jlong, std::shared_ptr<Splitter>> handle_map;
    static std::mutex handle_mutex;
    
    jlong registerSplitter(std::shared_ptr<Splitter> splitter) {
        std::lock_guard<std::mutex> lock(handle_mutex);
        jlong handle = reinterpret_cast<jlong>(splitter.get());
        handle_map[handle] = splitter;
        return handle;
    }
    
    std::shared_ptr<Splitter> getSplitter(jlong handle) {
        std::lock_guard<std::mutex> lock(handle_mutex);
        auto it = handle_map.find(handle);
        if (it == handle_map.end()) {
            return nullptr;
        }
        return it->second;
    }
    
    void releaseSplitter(jlong handle) {
        std::lock_guard<std::mutex> lock(handle_mutex);
        handle_map.erase(handle);  // shared_ptr 自动管理释放
    }
}

// close 函数
JNIEXPORT void JNICALL ..._close(JNIEnv *env, jobject, jlong splitter_addr) {
    JNI_FUNC_START
        HandleManager::releaseSplitter(splitter_addr);  // 安全释放
    JNI_FUNC_END_VOID(runtimeExceptionClass)
}
```

    for (int i = 0; i < 10; i++) {
        threads.emplace_back([&]() {
            HandleManager::unregisterHandle(handle);
        });
    }
    
    for (auto& t : threads) {
        t.join();
    }
    
    // 只有一次成功 unregister
    EXPECT_FALSE(HandleManager::isValidHandle(handle));
}

## 7. 附录

### 7.1 CWE-416 定义

> **CWE-416: Use After Free**
>
> The product reuses or references memory after it has been freed. At some point afterward, the memory may be allocated again and saved in another pointer, while the original pointer references a location somewhere within the new allocation. As operations are performed on the new allocation, the old pointer can be used to corrupt the new allocation.
>
> **Consequences**:
> - DoS: Crash, Exit, or Restart
> - Execute Code: Use after free often allows attackers to execute arbitrary code
> - Memory Corruption: Accessing freed memory may corrupt heap metadata

## 8. 结论

**JNI-013 是一个真实存在的 High 级别 Use After Free 漏洞**。

**关键风险**：
- delete 后未置 NULL，导致后续调用仍能获取相同地址
- NULL 检查基于地址值而非指针状态，失效
- 无 handle 状态管理，无法区分已释放和未释放的 handle
- 与 nativeClose (line 426) 同模式，需同步修复

**修复核心**：
1. **立即实施 Handle 注册表机制** - 追踪有效 handle，拒绝无效操作
2. **delete 后置 NULL** - 防御性编程
3. **Java 层增强同步** - 防止线程竞争
4. **长期重构为智能指针** - 彻底消除手动内存管理风险

**建议立即实施 P0 级别修复**，确保 native 层独立防御，不依赖 Java 层防护。
