# JNI-007 漏洞详细分析报告

## 1. 漏洞概述

### 基本信息
| 属性 | 值 |
|------|-----|
| **漏洞 ID** | JNI-007 |
| **类型** | Stack-based Buffer Overflow (CWE-121) |
| **严重性** | High |
| **置信度** | 85/100 |
| **状态** | CONFIRMED |
| **文件路径** | `cpp-omni/src/jni/deserializer.cpp` |
| **行号** | 74-77 |
| **函数名** | `Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_columnarShuffleParseBatch` |
| **信任边界** | JNI Interface (High Risk) - Spark JVM Process → Native C++ Library |

### 漏洞摘要
JNI 函数 `columnarShuffleParseBatch` 创建 VLA (Variable Length Array) `vecs[vecCount]`，其中 `vecCount` 直接从 protobuf 数据获取，未进行任何边界验证。恶意 protobuf 数据指定超大 `vecCount` 值可导致栈溢出，造成进程崩溃或潜在代码执行。

### 关联漏洞
- **JNI-008**: `rowShuffleParseBatch:174-178` - 同样 VLA 栈溢出模式
- **JNI-003**: `rowShuffleParseInit:127-136` - ParseFromArray 未检查返回值
- **VULN-CPP-JNI-002**: `columnarShuffleParseInit:26-36` - 相同 ParseFromArray 问题

## 2. 技术细节分析

### 2.1 漏洞代码

```cpp
// cpp-omni/src/jni/deserializer.cpp:68-77
JNIEXPORT void JNICALL
Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_columnarShuffleParseBatch(
    JNIEnv *env, jobject obj, jlong address, jintArray typeIdArray, jintArray precisionArray,
    jintArray scaleArray, jlongArray vecNativeIdArray)
{
    spark::VecBatch* vecBatch = reinterpret_cast<spark::VecBatch*>(address);
    int32_t vecCount = vecBatch->veccnt();        // ← 从 protobuf 获取，无验证
    int32_t rowCount = vecBatch->rowcnt();
    omniruntime::vec::BaseVector* vecs[vecCount]{};  // ← VLA 栈分配！危险！

    JNI_FUNC_START
    jint *typeIdArrayElements = env->GetIntArrayElements(typeIdArray, NULL);
    // ...
}
```

**关键问题**: VLA `vecs[vecCount]` 在栈上分配，vecCount 来自 protobuf 数据，无上限检查。

### 2.2 Protobuf 数据结构

```protobuf
// cpp-omni/src/proto/vec_data.proto:7-11
message VecBatch {
    int32 rowCnt = 1;
    int32 vecCnt = 2;   // ← int32，可为任意值，最大 2,147,483,647
    repeated Vec vecs = 3;
}
```

**问题**: `vecCnt` 定义为 `int32`，无任何约束，可被攻击者设置为任意值。

### 2.3 VLA 栈分配风险分析

| vecCount 值 | 栈空间需求 (64-bit) | 结果 |
|-------------|---------------------|------|
| 10,000 | 80 KB | 正常 |
| 100,000 | 800 KB | 可能正常 |
| 1,000,000 | 8 MB | 边界情况 |
| 10,000,000 | 80 MB | **栈溢出** |
| 100,000,000 | 800 MB | **栈溢出** |
| INT_MAX (2.1B) | 17 GB | **栈溢出** |

Linux 默认栈大小: ~8MB (`ulimit -s` 通常为 8192)

## 6. 缓解措施建议

### 6.1 立即修复

```cpp
JNIEXPORT void JNICALL
Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_columnarShuffleParseBatch(
    JNIEnv *env, jobject obj, jlong address, jintArray typeIdArray, jintArray precisionArray,
    jintArray scaleArray, jlongArray vecNativeIdArray)
{
    spark::VecBatch* vecBatch = reinterpret_cast<spark::VecBatch*>(address);
    int32_t vecCount = vecBatch->veccnt();
    int32_t rowCount = vecBatch->rowcnt();
    
    // === 新增：边界验证 ===
    constexpr int32_t MAX_VEC_COUNT = 10000;   // 业务合理上限
    constexpr int32_t MAX_ROW_COUNT = 1000000;
    
    if (vecCount <= 0 || vecCount > MAX_VEC_COUNT) {
        env->ThrowNew(runtimeExceptionClass, 
            "Invalid vecCount: must be between 1 and 10000");
        return;
    }
    
    if (rowCount <= 0 || rowCount > MAX_ROW_COUNT) {
        env->ThrowNew(runtimeExceptionClass,
            "Invalid rowCount: must be between 1 and 1000000");
        return;
    }
    
    // === 新增：使用 std::vector 替代 VLA ===
    std::vector<omniruntime::vec::BaseVector*> vecs(vecCount, nullptr);
    
    JNI_FUNC_START
    // ... 原有逻辑，使用 vecs.data() 或 vecs[i] 访问
}
```

## 8. 结论

**JNI-007 是真实存在的 High 级别漏洞**，需要立即修复。

**核心风险**:
- VLA `vecs[vecCount]` 基于 protobuf 数据，无上限检查
- 恶意 protobuf 可指定超大 vecCount 导致栈溢出
- 直接 JNI 入口 + 完全可控数据 = 高置信度漏洞

**修复核心**:
1. 添加 vecCount/rowCount 边界验证
2. 用 `std::vector` 替代 VLA（heap 分配）
3. 同步修复 JNI-008 相同模式

**CVSS 评分**: 7.5 (High)
