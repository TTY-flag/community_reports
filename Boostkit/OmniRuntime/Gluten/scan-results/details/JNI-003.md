# JNI-003 漏洞详细分析报告

## 1. 漏洞概述

### 基本信息
| 属性 | 值 |
|------|-----|
| **漏洞 ID** | JNI-003 |
| **类型** | Missing Input Validation (CWE-20) |
| **严重性** | Critical |
| **置信度** | 85/100 |
| **状态** | CONFIRMED |
| **文件路径** | `cpp-omni/src/jni/deserializer.cpp` |
| **行号** | 127-136 |
| **函数名** | `Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_rowShuffleParseInit` |
| **信任边界** | JNI Interface (High Risk) - Spark JVM Process → Native C++ Library |

### 漏洞摘要
JNI 函数 `rowShuffleParseInit` 中调用 protobuf 的 `ParseFromArray` 方法后，未检查返回值。当 protobuf 解析失败时，`ProtoRowBatch` 对象处于未定义状态（undefined state），但该对象指针仍被返回给 Java 端。后续函数访问此无效对象，可能导致：
- 程序崩溃（DoS）
- 内存访问违规
- 未定义行为（Undefined Behavior）
- 栈溢出（通过 VLA 分配）

### 关联漏洞
该漏洞与以下漏洞具有相同的根因模式：
- **JNI-007** (`columnarShuffleParseBatch:74-77`): VLA 栈溢出，使用未验证的 `vecCount`
- **JNI-008** (`rowShuffleParseBatch:174-178`): VLA 栈溢出，使用未验证的 `vecCount`
- **JNI-001** (`columnarShuffleParseInit:27-36`): 相同的 `ParseFromArray` 返回值未检查问题

### 2.2 Protobuf ParseFromArray 行为分析

根据 Google Protobuf 文档和源码：

```cpp
// protobuf message.h
bool Message::ParseFromArray(const void* data, int size);
```

**返回值语义**：
- 返回 `true`: 解析成功，对象处于有效状态
- 返回 `false`: 解析失败，对象可能处于以下状态：
  - **部分初始化**：部分字段已解析，其余为默认值
  - **完全未初始化**：所有字段为默认值（空对象）
  - **损坏状态**：内部指针可能指向无效内存

**解析失败的可能原因**：
1. 输入数据不是有效的 protobuf 格式
2. 输入数据损坏或不完整
3. 输入数据与 schema 不匹配
4. 输入数据包含未知字段（strict mode）
5. 输入数据长度不正确（length < 0 或 length 与实际数据不符）
6. 输入地址无效（address == 0 或指向非法内存）

### 2.3 ProtoRowBatch 数据结构


### 2.4 后续函数调用分析

返回的 `ProtoRowBatch` 指针会被以下 JNI 函数使用：

#### rowShuffleParseVecCount (行 150-157)
```cpp
JNIEXPORT jint JNICALL
Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_rowShuffleParseVecCount(
    JNIEnv *env, jobject obj, jlong address)
{
    JNI_FUNC_START
    spark::ProtoRowBatch* protoRowBatch = reinterpret_cast<spark::ProtoRowBatch*>(address);
    return protoRowBatch->veccnt();  // 调用未定义状态对象的方法
    JNI_FUNC_END(runtimeExceptionClass)
}
```

**风险**：
- 如果解析失败，`veccnt()` 返回默认值 0 或垃圾值
- 返回值被用于分配 VLA 数组 `vecs[vecCount]`
- 若 `vecCount` 为垃圾值（如负数转无符号后巨大），导致栈溢出

#### rowShuffleParseRowCount (行 160-167)
```cpp
JNIEXPORT jint JNICALL ...rowShuffleParseRowCount(...)
{
    spark::ProtoRowBatch* protoRowBatch = reinterpret_cast<spark::ProtoRowBatch*>(address);
    return protoRowBatch->rowcnt();  // 同样风险
    JNI_FUNC_END(runtimeExceptionClass)
}
```

#### rowShuffleParseBatch (行 170-220)
```cpp

{
    spark::ProtoRowBatch* protoRowBatch = reinterpret_cast<spark::ProtoRowBatch*>(address);

    for (; pos < vecCount; ++pos) {
```

**关键风险点**：
1. `vecs[vecCount]` VLA 分配 - 如果 `vecCount` 是垃圾值，栈溢出
2. `protoRowBatch->vectypes(pos)` - 越界访问，如果 `vecCount` > 实际 `vecTypes.size()`
3. `rows + offsets[i]` - 空数据访问 + 垃圾偏移，内存访问违规

### 2.5 JNI_FUNC_END 宏分析

```cpp
// cpp-omni/src/jni/jni_common.h:137-145
#define JNI_FUNC_START try {

#define JNI_FUNC_END(exceptionClass)                \
    }                                               \
    catch (const std::exception &e)                 \
    {                                               \
        env->ThrowNew(exceptionClass, e.what());    \
        return 0;                                   \
    }
```

**问题**：此宏只捕获 C++ `std::exception`，不会捕获：
- 内存访问违规（SIGSEGV）- 直接导致进程崩溃
- 未定义行为导致的任意结果
- Protobuf 内部解析异常（如果未抛出 std::exception）

### 3.4 攻击场景分析

#### 场景 1: 恶意 Executor 攻击

**前提条件**：
- Spark 集群被部分入侵，攻击者控制至少一个 Executor
- 或攻击者能提交恶意 Spark 任务

**攻击步骤**：
1. 在恶意 Executor 的 Map 任务中，构造格式错误的 Shuffle 数据
2. 数据通过 Shuffle 机制传输到其他 Executor
3. 目标 Executor 在 Reduce 任务中反序列化该数据
4. 调用 `rowShuffleParseInit`，解析失败返回无效对象
5. 后续调用导致崩溃或内存访问违规

**效果**：
- 目标 Executor 崩溃，任务失败
- 可能触发 Spark 任务重试机制
- 大规模攻击可导致集群资源耗尽（DoS）

#### 场景 2: 数据篡改攻击

**前提条件**：
- 网络传输层缺乏完整性保护（未加密）
- 或 Shuffle 服务（如 External Shuffle Service）被入侵

**攻击步骤**：
1. 在 Shuffle 数据传输过程中篡改数据
2. 破坏 protobuf 格式或注入恶意结构
3. 目标 Executor 解析时触发漏洞

#### 场景 3: 恶意输入数据

**前提条件**：
- 用户输入直接影响 Shuffle 数据内容
- 例如：用户提交的数据被序列化为 protobuf 格式

**攻击步骤**：
1. 用户提交特制数据
2. 数据在序列化过程中被处理
3. 如果序列化逻辑有缺陷，可能产生格式错误的数据
4. 反序列化时触发漏洞

2. **远程代码执行**: 理论上如果 protobuf 内部有其他漏洞，可能被串联利用（低概率）

### 6.1 立即修复建议

#### 修复代码

```cpp
// cpp-omni/src/jni/deserializer.cpp:127-136
JNIEXPORT jlong JNICALL
Java_com_huawei_boostkit_spark_serialize_ShuffleDataSerializerUtils_rowShuffleParseInit(
    JNIEnv *env, jobject obj, jlong address, jint length)
{
    JNI_FUNC_START
    
    // 1. 参数边界验证
    const int MAX_PROTOBUF_SIZE = 1024 * 1024 * 1024; // 1GB 上限
    if (length < 0 || length > MAX_PROTOBUF_SIZE) {
        env->ThrowNew(runtimeExceptionClass, "Invalid protobuf length parameter");
        return 0;
    }
    
    if (address == 0) {
        env->ThrowNew(runtimeExceptionClass, "Invalid address parameter (null pointer)");
        return 0;
    }
    
    // 2. 创建对象
    auto *protoRowBatch = new spark::ProtoRowBatch();
    
    // 3. 解析并检查返回值
    bool parseSuccess = protoRowBatch->ParseFromArray(
        reinterpret_cast<char*>(address), length);
    
    if (!parseSuccess) {
        // 4. 解析失败时清理资源并返回错误
        delete protoRowBatch;
        env->ThrowNew(runtimeExceptionClass, "Protobuf parse failed: invalid ProtoRowBatch data");
        return 0;
    }
    
    // 5. 可选：验证解析结果的有效性
    int32_t vecCount = protoRowBatch->veccnt();
    int32_t rowCount = protoRowBatch->rowcnt();
    
    // 验证 vecCount 和 rowCount 在合理范围内
    const int MAX_VEC_COUNT = 10000;  // 根据实际业务调整
    const int MAX_ROW_COUNT = 1000000;
    
    if (vecCount < 0 || vecCount > MAX_VEC_COUNT) {
        delete protoRowBatch;
        env->ThrowNew(runtimeExceptionClass, "Invalid vecCount in ProtoRowBatch");
        return 0;
    }
    
    if (rowCount < 0 || rowCount > MAX_ROW_COUNT) {
        delete protoRowBatch;
        env->ThrowNew(runtimeExceptionClass, "Invalid rowCount in ProtoRowBatch");
        return 0;
    }
    
    // 6. 验证 vecTypes 数量与 vecCount 匹配
    if (protoRowBatch->vectypes_size() != vecCount) {
        delete protoRowBatch;
        env->ThrowNew(runtimeExceptionClass, "vecTypes count mismatch with vecCount");
        return 0;
    }
    
    return (jlong)(protoRowBatch);
    JNI_FUNC_END(runtimeExceptionClass)
}
```

### 6.4 架构改进建议

#### 使用安全的 protobuf 解析工具

参考项目中已有的正确实现（`cpp/core/compute/ProtobufUtils.cc`）：

```cpp
bool parseProtobuf(const uint8_t* buf, int bufLen, google::protobuf::Message* msg) {
  google::protobuf::io::CodedInputStream codedStream{buf, bufLen};
  codedStream.SetRecursionLimit(100000);  // 设置递归限制
  return msg->ParseFromCodedStream(&codedStream);  // 返回值明确返回
}
```

```cpp
template<typename ProtoType>
static ProtoType* safeParseProtobuf(
    JNIEnv* env, jlong address, jint length, const char* errorMessage) {
    
    if (!validateParseParams(env, address, length)) {
        return nullptr;
    }
    
    auto* obj = new ProtoType();
    if (!obj->ParseFromArray(reinterpret_cast<char*>(address), length)) {
        delete obj;
        env->ThrowNew(runtimeExceptionClass, errorMessage);
        return nullptr;
    }
    
    return obj;
}
```

## 8. 结论

**JNI-003 是一个真实存在的 Critical 级别漏洞**，需要立即修复。

**关键风险**：
- 攻击者可以通过控制 Shuffle 数据内容触发漏洞
- 导致 Executor 进程崩溃（DoS）
- 可能串联其他漏洞造成更严重后果

**修复核心**：
1. 检查 `ParseFromArray` 返回值
2. 添加参数边界验证
3. 解析失败时正确清理资源

**建议立即实施 P0 级别修复**，然后逐步完善防护措施。
