## 2. 根本原因分析

### 2.1 问题代码

```cpp
// 第 159-167 行
if (selected_instance) {
    // 找到selected_instance所在的组
    for (auto &group_info : group_infos) {  // ← 错误：遍历所有组
        // 这里需要根据实例找到组ID，可能需要额外的映射
        // 简化处理：假设我们可以通过某种方式知道实例的组ID
        // 在实际实现中，可能需要维护实例到组的映射
        selected_per_group[group_info.group_id]++;  // ← 所有组都增加计数
    }
}
```

### 2.2 逻辑错误详解

**预期行为**：当选中一个实例后，应该只增加该实例所属组的 `selected_per_group` 计数。

**实际行为**：代码遍历所有 `group_infos`，对**每个组**都增加计数，导致所有组的负载计数同步增长。

**正确实现应该是**：
```cpp
if (selected_instance) {
    selected_per_group[selected_instance->group_id]++;
}
```

每个 `Instance` 对象包含 `group_id` 属性（见第 28 行 `AddInstance` 函数签名），可直接通过 `selected_instance->group_id` 获取正确的组 ID。

### 2.3 代码注释暴露的问题

注释中写道：
```
// 这里需要根据实例找到组ID，可能需要额外的映射
// 简化处理：假设我们可以通过某种方式知道实例的组ID
```

这表明开发者**明知实现有问题**，但采用了错误的"简化处理"。实际上 `Instance` 结构体已包含 `group_id`，完全不需要额外映射。

### 4.3 攻击者利用可行性分析

| 分析维度 | 结论 |

**具体分析**：

1. **无法触发特定行为**：负载均衡是内部算法，攻击者发送的推理请求不会影响 `SelectSingleInstance` 的具体执行逻辑。

2. **无法预测或利用结果**：即使知道负载均衡失效，攻击者也无法：
   - 强制请求路由到特定设备
   - 导致特定设备过载

3. **无法造成 DoS**：虽然负载均衡失效，但系统仍能正常处理请求，不会拒绝服务。

## 5. 利用场景分析

### 5.1 理论上可能的间接影响场景

**场景：资源耗尽导致服务降级**

1. 系统有多个 NPU 设备（group），负载均衡失效后
2. 某个设备持续被选择（因随机选择逻辑）
3. 该设备处理大量请求，可能达到硬件极限
4. 其他设备完全空闲

**实际可能性评估**：

| 因素 | 分析 |
|------|------|
| 设备选择随机性 | `FindCandidateGroups` 找到最小负载组后，组间选择有一定随机性 |
| 并发请求量 | 高并发场景下，某设备可能被多次选中 |
| 硬件极限 | NPU 设备有处理能力上限，但通常有排队机制 |

**结论**：虽然理论上可能导致资源不均衡，但实际影响取决于并发量和设备数量。

### 5.2 攻击者无法主动利用的原因

攻击者发送推理请求时：

1. **无法指定目标设备**：请求由调度器自动分配
2. **无法观察调度结果**：攻击者只能获得推理结果，不知道哪个设备处理了请求
3. **无法测量负载分布**：无法探测内部负载状态

因此，攻击者**无法主动触发或利用此缺陷**。

## 6. 修复建议

### 6.1 直接修复方案

**修改 `src/scheduler.cpp` 第 159-167 行**：

```cpp
// 修复后的代码
if (selected_instance) {
    // 直接使用选中实例的 group_id
    selected_per_group[selected_instance->group_id]++;
}
```

**修复说明**：
- 删除错误的全组遍历循环
- 直接使用 `selected_instance->group_id` 更新正确的组计数
- 每个 `Instance` 对象包含 `group_id` 属性（见第 28 行）

### 6.2 验证修复正确性

修复后，负载均衡逻辑将正确工作：

1. `CalculateGroupLoads()` 计算当前运行实例数
2. `selected_per_group` 正确跟踪已选中的实例数
3. `FindCandidateGroups()` 能正确找到负载最小的组
4. 组间选择基于真实负载差异

### 6.3 附加建议

1. **添加单元测试**：验证多组场景下负载均衡的正确性
2. **添加日志监控**：记录各组负载分布，便于检测异常
3. **代码审查**：检查其他类似"简化处理"注释的代码段
