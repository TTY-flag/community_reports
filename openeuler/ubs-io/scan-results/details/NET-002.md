# NET-002: Untrusted Pointer Dereference

## 漏洞概述

| 属性 | 值 |
|------|-----|
| **漏洞ID** | NET-002 |
| **CWE** | CWE-822 (Untrusted Pointer Dereference) |
| **严重性** | HIGH |
| **置信度** | 90 |
| **模块** | net |
| **文件** | `ubsio-boostio/src/net/net_engine.h:794-795` |
| **函数** | `SyncCall` |

## 漏洞描述

RDMA 网络响应中的地址字段 (`respMsg.address`) 被直接通过 `reinterpret_cast` 转换为指针并使用，没有任何验证。恶意对端可以提供任意地址值，导致：
- **进程崩溃**：解引用无效地址导致 SIGSEGV
- **内存破坏**：写入到任意内存位置
- **信息泄露**：读取任意内存区域

## 漏洞代码

```cpp
// net_engine.h:794-795
*resp = reinterpret_cast<TResp *>(respMsg.address);
respLen = respMsg.size;
return BIO_OK;
```

**问题分析**：
1. `respMsg.address` 来自网络响应，完全由对端控制
2. 直接 `reinterpret_cast` 将网络数据当作本地指针
3. `respLen = respMsg.size` 同样未验证，可能导致越界读取

## 攻击场景构造

### 场景 1: 进程崩溃 (DoS)

**攻击者**：拥有 RDMA 网络访问权限的恶意节点

**步骤**：
1. 连接到目标 UBS-IO 服务器的 RDMA 端口
2. 发送 RPC 请求（如 `Bio::Put` 或 `Bio::Get`）
3. 构造响应消息：
   ```c
   respMsg.address = 0x00000000;  // NULL 指针
   respMsg.size = 1024;
   respMsg.errorCode = BIO_OK;
   ```
4. 目标服务器解引用 NULL 指针 → SIGSEGV → 服务崩溃

### 场景 2: 内存信息泄露

**步骤**：
1. 构造响应指向已知内存区域：
   ```c
   respMsg.address = 0x7ffff7dd0000;  // 指向 libc 数据区
   respMsg.size = 4096;
   ```
2. 调用方读取响应数据时，泄露敏感内存内容（密钥、凭证等）

### 场景 3: 内存破坏 (写入攻击)

**条件**：如果调用方写入到响应指针

**步骤**：
1. 构造响应指向关键内存结构：
   ```c
   respMsg.address = &target_process->credentials;
   ```
2. 调用方写入伪造数据 → 权限提升或执行流劫持

| 维度 | 评估 |
|------|------|
| **可触发性** | HIGH - RDMA 对端完全可控 |
| **攻击者要求** | 网络访问 + RDMA 连接能力 |
| **影响范围** | HIGH - 整个服务进程 |

## 修复建议

### 方案 1: 响应地址验证 (推荐)

```cpp
// net_engine.h:794-796 修复版本
template <typename TResp>
NetResult SyncCall(uint16_t opCode, TReq &req, ChannelPtr &ch, TResp **resp, uint32_t &respLen)
{
    // ... existing code ...
    
    if (UNLIKELY(respMsg.errorCode != BIO_OK)) {
        return respMsg.errorCode;
    }
    
    // ===== 新增验证 =====
    // 1. 检查地址范围合法性
    if (respMsg.address == nullptr || respMsg.size == 0) {
        NET_LOG_ERROR("Invalid response: null address or zero size");
        return BIO_ERR_INVALID_RESPONSE;
    }
    
    // 2. 检查地址是否在进程合法内存范围内
    // 使用 mincore 或查询进程内存映射验证
    if (!IsValidMemoryRange(respMsg.address, respMsg.size)) {
        NET_LOG_ERROR("Response address outside valid memory range: " << respMsg.address);
        return BIO_ERR_INVALID_RESPONSE;
    }
    
    // 3. 检查大小是否合理（不超过最大响应限制）
    const uint32_t MAX_RESPONSE_SIZE = 16 * 1024 * 1024; // 16MB
    if (respMsg.size > MAX_RESPONSE_SIZE) {
        NET_LOG_ERROR("Response size exceeds limit: " << respMsg.size);
        return BIO_ERR_INVALID_RESPONSE;
    }
    
    *resp = reinterpret_cast<TResp *>(respMsg.address);
    respLen = respMsg.size;
    return BIO_OK;
}

// 辅助函数
bool IsValidMemoryRange(void* addr, size_t size) {
    // 检查地址是否在栈、堆或映射区域范围内
    uintptr_t ptr = (uintptr_t)addr;
    
    // 检查是否在用户空间范围内
    if (ptr < 0x10000 || ptr > 0x7fffffffffff) {
        return false;
    }
    
    // 可选：使用 mincore() 检查页面是否映射
    unsigned char vec;
    if (mincore(addr, size, &vec) == -1) {
        return false;
    }
    
    return true;
}
```

### 方案 2: 使用安全缓冲区 (替代方案)

不直接使用网络地址，而是分配本地缓冲区并复制数据：

```cpp
// 使用本地分配的缓冲区替代直接指针
TResp *localResp = new TResp();
if (respMsg.size <= sizeof(TResp)) {
    memcpy(localResp, respMsg.address, respMsg.size);
    *resp = localResp;
    respLen = respMsg.size;
} else {
    NET_LOG_ERROR("Response size mismatch");
    delete localResp;
    return BIO_ERR_INVALID_RESPONSE;
}
```

### 方案 3: RDMA 层面验证

在 RDMA 协议层添加完整性验证：

```cpp
// 使用 HMAC 或签名验证响应完整性
if (!VerifyResponseIntegrity(respMsg)) {
    return BIO_ERR_RESPONSE_CORRUPTED;
}
```

## 缓解措施

### 立即缓解 (无需代码修改)

1. **网络隔离**：将 RDMA 端口限制在可信网络范围内
2. **防火墙规则**：仅允许已知节点连接
3. **监控告警**：检测异常响应模式（NULL 地址、超大 size）

### 短期缓解

1. 在 SDK 层添加二次验证
2. 添加响应大小硬限制
3. 使用沙箱/容器隔离服务进程
