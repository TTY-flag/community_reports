# DF-010: ShmCreate 缺少服务端 size/quota 校验导致共享内存资源耗尽

## 漏洞概述

| 属性 | 值 |
|------|-----|
| **漏洞ID** | DF-010 |
| **类型** | Missing Size Validation |
| **CWE** | CWE-190 (Integer Overflow or Wraparound) |
| **严重性** | High (原 Medium，验证后提升) |
| **置信度** | 85 |
| **文件** | src/mxm_shm/ipc_handler.cpp |
| **行号** | 118-231 |
| **函数** | ShmCreate |
| **模块** | mxm_shm, under_api |

## Codex 二次确认补充

- 结论：属实，`ShmCreate` 服务端缺少 `size_` 上限和 quota 校验；更偏 CWE-400，整数溢出只是潜在下游风险。
- 场景：需要能访问 shm IPC；客户端 SDK 校验可被 raw IPC 绕过，所以服务端仍需兜底。
- 本段为二次确认补充，原报告其他内容保留不变。
## 漏洞描述

`ShmCreate` 函数创建共享内存时，接受来自 IPC 请求的 `size_` 参数，但**缺乏上限验证**：
- `request->size_` 直接赋值给 `createParam.size`（Line 204）
- 未检查是否超出系统可用内存
- 未检查是否超出合理分配上限
- 仅验证 `regionDesc.num`（节点数量），忽略 size 字段

该漏洞与 DF-001 类似，但针对共享内存创建场景。攻击者可：
1. **资源耗尽**：消耗系统共享内存资源
2. **整数溢出**：超大 size 导致后续计算错误
3. **拒绝服务**：阻止其他进程创建共享内存


## 影响评估

### 直接影响

| 影响 | 描述 | 严重性 |
|------|------|--------|
| **共享内存耗尽** | 系统 shm 资源被消耗 | ⚠️ High |
| **其他进程受阻** | 无法创建新的共享内存 | ⚠️ High |
| **系统不稳定** | 可能触发 OOM 或系统异常 | ⚠️ High |
| **整数溢出风险** | size 参数在后续计算中可能溢出 | ⚠️ Medium |

### 与 DF-001 的区别

| 漏洞 | 资源类型 | 影响 | 验证状态 |
|------|----------|------|----------|
| DF-001 | 内存分配 (malloc) | 进程内存耗尽 | 缺失 size 验证 |
| DF-010 | 共享内存 (shm) | 系统级共享内存耗尽 | 缺失 size 验证 |

共享内存是**系统级资源**，影响范围更大：

- 跨进程共享
- 持久化存储（即使创建进程退出）
- 可能影响整个集群的共享内存管理

## 修复建议

### 立即修复方案

在 `ShmCreate` 中添加 size 上限验证：

```cpp
// src/mxm_shm/ipc_handler.cpp:118
int MxmServerMsgHandle::ShmCreate(const MsgBase* req, MsgBase* rsp, const MxmComUdsInfo& udsInfo)
{
    auto request = dynamic_cast<const ShmCreateRequest*>(req);
    auto response = dynamic_cast<CommonResponse*>(rsp);
    if (request == nullptr || response == nullptr) {
        return MXM_ERR_NULLPTR;
    }

    // ===== 新增: Size 上限验证 =====
    const size_t MAX_SHM_SIZE = 1024ULL * 1024ULL * 1024ULL; // 1GB
    
    if (request->size_ == 0) {
        DBG_LOGERROR("Invalid size: 0 for shared memory");
        response->errCode_ = MXM_ERR_PARAM_INVALID;
        return MXM_ERR_PARAM_INVALID;
    }
    
    if (request->size_ > MAX_SHM_SIZE) {
        DBG_LOGERROR("Size exceeds maximum limit for shm: size=" << request->size_ 
                     << ", max=" << MAX_SHM_SIZE << ", name=" << request->name_);
        DBG_AUDITERROR("SECURITY: Oversized shm create rejected. "
                       << "uid=" << udsInfo.uid << ", size=" << request->size_);
        response->errCode_ = MXM_ERR_SIZE_TOO_LARGE;
        return MXM_ERR_SIZE_TOO_LARGE;
    }
    // ===== 验证结束 =====

    // 原有节点数量验证
    if (request->regionDesc_.num > MEM_TOPOLOGY_MAX_HOSTS) {
        ...
    }
    
    // 原有逻辑继续
    createParam.name = request->name_;
    createParam.size = request->size_;
    ...
}
```

### 动态限制方案

```cpp
// 根据系统共享内存可用空间动态限制
size_t GetAvailableShmSpace() {
    // 检查 /proc/sys/kernel/shmmax 或系统配置
    // 或通过 sysinfo 获取可用内存
    struct sysinfo info;
    if (sysinfo(&info) == 0) {
        return info.freeram * info.mem_unit;
    }
    return MAX_SHM_SIZE;  // 默认值
}

// 在 ShmCreate 中使用
auto available = GetAvailableShmSpace();
auto maxAllowed = std::min(MAX_SHM_SIZE, available / 2);  // 不超过可用空间的50%
if (request->size_ > maxAllowed) {
    return MXM_ERR_INSUFFICIENT_MEMORY;
}
```

### 长期改进方案

1. **用户配额系统**：
   - 每个用户的共享内存总上限
   - 在 RegionManager 或 ShmManager 中跟踪

2. **系统配额集成**：
   - 与 Linux `/proc/sys/kernel/shmmax` 配置联动
   - 与 cgroup 内存限制联动

3. **配额管理接口**：

```cpp
// 配额管理类
class ShmQuotaManager {
public:
    bool CheckUserQuota(uint32_t uid, size_t requestSize) {
        auto currentUsage = GetUserTotalShmSize(uid);
        auto quota = GetUserShmQuota(uid);
        return (currentUsage + requestSize) <= quota;
    }
    
    size_t GetUserShmQuota(uint32_t uid) {
        // root 用户无限制
        if (uid == 0) return SIZE_MAX;
        // 普通用户限制
        return DEFAULT_USER_QUOTA;  // 如 256MB
    }
};
```
