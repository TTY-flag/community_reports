# DF-001: AppMallocMemory 缺少服务端 size/quota 校验导致资源耗尽

## 漏洞概述

| 属性 | 值 |
|------|-----|
| **漏洞ID** | DF-001 |
| **类型** | Integer Overflow / Resource Exhaustion |
| **CWE** | CWE-190 (Integer Overflow or Wraparound) |
| **严重性** | High |
| **置信度** | 85 |
| **文件** | src/mxm_lease/ipc_handler.cpp |
| **行号** | 169-203 |
| **函数** | AppMallocMemory |
| **模块** | mxm_lease |

## Codex 二次确认补充

- 结论：属实，但更准确地说是服务端缺少 size/quota 校验导致的资源耗尽风险；整数溢出目前证据不足。
- 场景：需要本机可访问 ubs-mem IPC 的进程，通常是 `ubsmd` 组用户/进程；不是默认任意本地用户。
- 本段为二次确认补充，原报告其他内容保留不变。
## 漏洞描述

`AppMallocMemory` 函数接受来自 IPC 请求的 `size_` 参数，但**缺乏上限验证**。该参数直接传递给底层内存分配函数 (`LeaseMalloc` → `pUbseMemFdCreateWithCandidate` / `pUbseMemNumaCreateWithCandidate`)，未检查是否超出系统可用内存或合理分配上限。

攻击者可通过构造超大 size 请求导致：
1. **资源耗尽**：消耗所有可用内存，影响系统稳定性
2. **整数溢出**：在后续计算中可能触发整数溢出（如缓存计算）
3. **拒绝服务**：阻止其他合法进程获取内存资源

## 触发条件

### 前置条件

| 条件 | 说明 |
|------|------|
| **IPC Socket 访问** | 需能连接到 `/var/run/ubs_mem/mem_lease` |
| **本地进程** | 必须是本地进程（Unix Domain Socket） |
| **无特权要求** | 任何本地用户均可调用（无权限检查） |

### 攻击者需要满足的条件

1. **无需特权**：任何本地用户（uid != 0）均可发起请求
2. **无需特定身份**：udsInfo.uid/gid/pid 用于审计日志，不用于权限验证
3. **仅需 IPC 连接**：通过 mxm_lease IPC 库连接即可

### 攻击步骤

```cpp
// 攻击代码示例
#include "mxmem_lease.h"

int main() {
    // 1. 初始化 IPC 连接
    auto lease = MxMemLease::GetInstance();
    
    // 2. 构造恶意请求 - 超大 size
    AppMallocMemoryRequest request;
    request.size_ = UINT64_MAX;  // 或超大值如 100GB
    request.regionName_ = "default";
    request.isNuma_ = false;
    
    // 3. 发送 IPC 消息
    lease->MallocMemory(request);  // 触发资源耗尽
    
    return 0;
}
```

## 影响评估

### 直接影响

| 影响 | 描述 |
|------|------|
| **资源耗尽** | 系统内存被大量分配，导致其他进程无法获取内存 |
| **拒绝服务 (DoS)** | 服务进程可能因内存不足而崩溃或挂起 |
| **系统不稳定** | 可能触发 OOM Killer，影响关键系统服务 |

### 间接影响

1. **整数溢出风险**：
   - `mls_manager.cpp:230` 存在计算: `size > (UINT64_MAX - info.unitSize)`
   - 若 size 接近 UINT64_MAX，可能绕过此检查
   
2. **缓存系统干扰**：
   - 超大内存分配可能干扰缓存策略
   - `ReuseCachedLeaseMemory` 逻辑可能被利用

### CVSS-like 评分因素

| 因素 | 值 | 说明 |
|------|-----|------|
| Attack Vector | Local | 本地 IPC 攻击 |
| Attack Complexity | Low | 无特殊条件 |
| Privileges Required | Low | 任意本地用户 |
| User Interaction | None | 无需交互 |
| Scope | Changed | 影响其他进程 |
| Impact | High | DoS + 系统稳定性 |

## 修复建议

### 立即修复方案

在 `AppMallocMemory` 中添加 size 上限验证：

```cpp
// src/mxm_lease/ipc_handler.cpp:169
int MxmServerMsgHandle::AppMallocMemory(const MsgBase* req, MsgBase* rsp, const MxmComUdsInfo& udsInfo)
{
    if (req == nullptr || rsp == nullptr) {
        return MXM_ERR_NULLPTR;
    }
    auto request = dynamic_cast<const AppMallocMemoryRequest*>(req);
    auto response = dynamic_cast<AppMallocMemoryResponse*>(rsp);
    if (request == nullptr || response == nullptr) {
        return MXM_ERR_NULLPTR;
    }

    // ===== 新增: Size 上限验证 =====
    // 定义最大单次分配限制 (例如: 1GB 或系统可用内存的合理比例)
    const size_t MAX_SINGLE_ALLOC_SIZE = 1024ULL * 1024ULL * 1024ULL; // 1GB
    
    if (request->size_ == 0) {
        DBG_LOGERROR("Invalid size: 0");
        response->errCode_ = MXM_ERR_PARAM_INVALID;
        return MXM_ERR_PARAM_INVALID;
    }
    
    if (request->size_ > MAX_SINGLE_ALLOC_SIZE) {
        DBG_LOGERROR("Size exceeds maximum limit: size=" << request->size_ 
                     << ", max=" << MAX_SINGLE_ALLOC_SIZE);
        response->errCode_ = MXM_ERR_SIZE_TOO_LARGE;
        return MXM_ERR_SIZE_TOO_LARGE;
    }
    
    // 可选: 检查系统可用内存
    // auto availableMem = GetSystemAvailableMemory();
    // if (request->size_ > availableMem * 0.5) { // 不超过可用内存的50%
    //     return MXM_ERR_INSUFFICIENT_MEMORY;
    // }
    // ===== 验证结束 =====

    // 原有逻辑继续...
    auto ret = BorrowMemory(*request, *response, udsInfo);
    ...
}
```

### 配套修复

1. **在 mx_def.h 中定义常量**：

```cpp
// src/include/mx_def.h
#define MAX_SINGLE_ALLOC_SIZE (1024ULL * 1024ULL * 1024ULL) // 1GB
#define MAX_USER_TOTAL_ALLOC_SIZE (4ULL * 1024ULL * 1024ULL * 1024ULL) // 4GB per user
```

2. **添加错误码**：

```cpp
// src/include/mx_error.h
#define MXM_ERR_SIZE_TOO_LARGE  (-1001)
#define MXM_ERR_INSUFFICIENT_MEMORY (-1002)
```

3. **在 BorrowNewMemory 中也添加验证**：

```cpp
// src/mxm_lease/ipc_handler.cpp:102
static int BorrowNewMemory(const AppMallocMemoryRequest& request, ...)
{
    // 双重验证（防御性编程）
    if (request.size_ > MAX_SINGLE_ALLOC_SIZE) {
        return MXM_ERR_SIZE_TOO_LARGE;
    }
    ...
}
```

### 长期改进方案

1. **用户配额系统**：
   - 为每个 uid 设置累计分配上限
   - 在 MLSManager 中跟踪用户总分配量
   
2. **动态限制**：
   - 根据系统可用内存动态调整上限
   - 实现类似 cgroup 的内存控制

3. **速率限制**：
   - 限制单用户单位时间内的分配请求次数
   - 防止恶意快速多次分配

TEST(AppMallocMemory, RejectLargeSize) {

```cpp
TEST(AppMallocMemory, AcceptNormalSize) {
    AppMallocMemoryRequest request;

}
```
