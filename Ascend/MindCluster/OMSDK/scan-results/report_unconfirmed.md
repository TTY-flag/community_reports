## 3. Low 漏洞 (3)

### [VULN-SEC-AUTH-001] ip_header_trust - token_required

**严重性**: Low（原评估: High → 验证后: Low） | **CWE**: CWE-287 | **置信度**: 55/100 | **状态**: POSSIBLE | **来源**: security-auditor

**位置**: `src/om/platform/MindXOM_SDK/src/app/sys/RedfishServer/token_auth.py:58-72` @ `token_required`
**模块**: user_manager

**描述**: IP地址来源于HTTP请求头X-Real-Ip，可能被攻击者伪造。如果服务器未正确配置反向代理信任机制，攻击者可以通过伪造此头绕过IP相关的安全检查（如登录规则、会话验证）。

**漏洞代码** (`src/om/platform/MindXOM_SDK/src/app/sys/RedfishServer/token_auth.py:58-72`)

```c
try:
    request_ip = request.headers["X-Real-Ip"]
except Exception:
    run_log.error("Get data from request header failed")
    request_ip = None
```

**达成路径**

request.headers["X-Real-Ip"] → request_ip → IPChecker.check → TokenVerification.get_all_info

**验证说明**: IP address from X-Real-Ip header can be forged if reverse proxy trust is misconfigured. IPChecker validates format but attacker can still spoof IP if proxy doesn't strip the header. This affects IP-based login rules/session validation. Not in exclusion scope (not authentication bypass chain, but implementation issue). Mitigation exists (IPChecker) reducing severity.

**评分明细**: base: 30 | reachability: 20 | controllability: 15 | mitigations: -15 | context: 0 | cross_file: 5

### [VULN-SEC-CERT-001] certificate_content_handling - get_ssl_ctx_by_cert_crl

**严重性**: Low（原评估: Medium → 验证后: Low） | **CWE**: CWE-295 | **置信度**: 55/100 | **状态**: POSSIBLE | **来源**: security-auditor

**位置**: `src/om/src/app/sys_om/RedfishServer/om_event_subscription/subscription_client.py:55-66` @ `get_ssl_ctx_by_cert_crl`
**模块**: om_event_subscription

**描述**: 用户提供的证书内容直接写入临时文件并用于创建SSL上下文。虽然使用了权限控制(0o600)，但需要验证证书内容是否经过充分校验，防止恶意证书注入。

**漏洞代码** (`src/om/src/app/sys_om/RedfishServer/om_event_subscription/subscription_client.py:55-66`)

```c
with os.fdopen(os.open(tmp_cert_file, flags=flags, mode=0o600), "w") as tmp_file:
    tmp_file.write(cert_with_crl.cert_contents)
...
with os.fdopen(os.open(tmp_crl_file, flags=flags, mode=0o600), "w") as tmp_file:
    tmp_file.write(cert_with_crl.crl_contents)
```

**达成路径**

cert_with_crl.cert_contents → tmp_cert_file → TlsConfig.get_client_ssl_context

**验证说明**: User-provided certificate content written directly to temp file. While file permissions (0o600) are secure, certificate content validation relies on downstream SSL context creation. Malformed PEM could cause issues. File path is hardcoded (/run/web/tmp_subs_crt.crt) preventing path traversal. Not in exclusion scope (not TLS verification itself, but file handling). Low severity due to format constraint.

**评分明细**: base: 30 | reachability: 20 | controllability: 10 | mitigations: -5 | context: 0 | cross_file: 0
