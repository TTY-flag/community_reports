# DF-008：RPC请求响应包含敏感信息致信息泄露

## 1. 漏洞概述

### 基本信息
| 属性 | 值 |
|------|-----|
| **漏洞ID** | DF-008 |
| **漏洞类型** | 信息泄露 (Information Disclosure) |
| **CWE分类** | CWE-200: Exposure of Sensitive Information to an Unauthorized Actor |
| **严重性** | 中等 (Medium) |
| **CVSS评分** | 5.3 (AV:N/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N) |

### 漏洞位置
| 项目 | 详情 |
|------|------|
| **文件路径** | `/dynolog_npu/dynolog/src/rpc/SimpleJsonServer.cpp` |
| **行号** | 322 |
| **函数名** | `SimpleJsonServerBase::processOne()` |
| **代码片段** | `LOG(INFO) << "RPC message received = " << request_str;` |

### 漏洞描述
RPC请求处理过程中，服务端将完整的请求内容（包含敏感业务数据）通过 `LOG(INFO)` 级别记录到日志文件，可能导致敏感信息泄露到持久化日志中，违反安全最佳实践。

---

## 2. 漏洞触发条件

### 触发路径
```
攻击者 → TCP连接(端口1778) → SSL_accept(可选) → get_message() → request_str → LOG(INFO) → 日志文件
```

### 触发步骤
1. **网络连接**: 攻击者向目标服务器TCP端口1778建立连接
2. **协议交互**: 发送符合dynolog RPC协议的消息（格式: `[int32长度][JSON内容]`）
3. **自动触发**: 服务端接收消息后，在 `processOne()` 函数第322行自动将完整请求记录到日志
4. **信息泄露**: 日志被写入磁盘，可能包含敏感业务配置信息

### 前置条件
| 条件 | 说明 |
|------|------|
| **网络可达** | 攻击者需能够访问目标服务器的1778端口 |
| **服务运行** | dynolog守护进程必须正在运行并监听端口 |
| **无认证绕过** | 如果配置为NO_CERTS模式，无需任何认证；如果有TLS证书认证，需有效客户端证书 |

### 触发门槛分析
- **低门槛**: 在NO_CERTS模式下，任何能访问网络的攻击者即可触发
- **中门槛**: 在TLS模式下，需持有有效客户端证书（但日志泄露仍会发生）

## 6. 修复建议

### 修复方案1: 移除敏感信息日志记录 (推荐)

**修改位置**: `SimpleJsonServer.cpp:322`

**修复代码**:
```cpp
void SimpleJsonServerBase::processOne() noexcept {
    LOG(INFO) << "Waiting for connection.";
    ClientSocketWrapper client;
    if (!client.accept(sock_fd_, ctx_)) {
        return;
    }
    std::string request_str = client.get_message();
    
    // 修复: 不记录完整请求内容，仅记录请求类型或摘要信息
    // 原代码: LOG(INFO) << "RPC message received = " << request_str;
    
    // 方案A: 仅记录请求长度（无敏感信息）
    LOG(INFO) << "RPC message received, length = " << request_str.size() << " bytes";
    
    // 方案B: 仅记录函数名（需先解析JSON）
    // 注意: 这需要先解析JSON，可能影响性能
    // auto request_json = toJson(request_str);
    // if (!request_json.empty() && request_json.contains("fn")) {
    //     LOG(INFO) << "RPC request type: " << request_json["fn"];
    // }
    
    auto response_str = processOneImpl(request_str);
    ...
}
```

### 修复方案2: 降低日志级别

**修改代码**:
```cpp
// 将INFO级别改为DEBUG级别，生产环境通常不记录DEBUG日志
LOG(DEBUG) << "RPC message received = " << request_str;
```

**注意**: 此方案不彻底，DEBUG日志在某些调试场景仍可能被启用。

### 修复方案3: 敏感字段脱敏

**修改代码**:
```cpp
std::string sanitize_request(const std::string& request) {
    // 解析JSON并脱敏敏感字段
    try {
        auto json = nlohmann::json::parse(request);
        if (json.contains("config")) {
            json["config"] = "[REDACTED]";
        }
        if (json.contains("log_file")) {
            json["log_file"] = "[REDACTED]";
        }

            json["pids"] = "[REDACTED]";
        }
        return json.dump();
    } catch (...) {
        return "[parse error - content redacted]";
    }
}

// 使用脱敏后的内容记录日志
LOG(INFO) << "RPC message received = " << sanitize_request(request_str);
```

### 修复方案4: 配置化日志级别控制

**添加配置选项**:
```cpp

// 在SimpleJsonServer.cpp中使用

```

### 最佳修复方案组合

推荐同时实施以下措施：
1. **立即修复**: 移除完整请求日志（方案1）
2. **长期改进**: 建立敏感数据日志策略，审计所有LOG语句
3. **配置加固**: 生产环境默认禁用DEBUG级别日志
4. **代码规范**: 制定日志记录安全规范，禁止记录完整外部输入

修复后应验证：
