## CWE-59 漏洞解析

### CWE 定义

CWE-59 描述的是在访问文件之前没有正确解析或验证符号链接（symlink）或硬链接，可能导致：

1. **符号链接跟随攻击**: 攻击者创建符号链接指向敏感文件，程序跟随链接访问敏感文件
2. **TOCTOU (Time-of-Check to Time-of-Use)**: 检查和操作之间的时间窗口内，攻击者替换链接目标
3. **权限绕过**: 通过符号链接绕过权限检查

### 相关攻击类型

| 攻击类型 | 描述 |
|----------|------|
| Symlink Following | 符号链接跟随，访问攻击者指定的目标 |
| Hardlink Attack | 硬链接攻击，替换文件内容 |
| Directory Traversal via Symlink | 通过符号链接进行目录遍历 |
| Privilege Escalation via Symlink | 利用符号链接提升权限 |

## 漏洞利用分析

### 1. 符号链接攻击场景

**前提条件**:
- 工具以更高权限运行（sudo/setuid）
- 攻击者可以创建符号链接

**攻击步骤**:

```
步骤 1: 攻击者创建指向敏感程序的符号链接
─────────────────────────────────────────────────────
$ ln -s /bin/su /tmp/fake_analysis_tool
# /bin/su 是 setuid 程序，可以提升到 root

步骤 2: 诱导执行符号链接
─────────────────────────────────────────────────────
$ sudo msmemscope /tmp/fake_analysis_tool

步骤 3: realpath() 解析
─────────────────────────────────────────────────────
realpath("/tmp/fake_analysis_tool") → "/bin/su"
# realpath 正确解析了符号链接，但没有阻止执行 setuid 程序

步骤 4: execvpe() 执行
─────────────────────────────────────────────────────
execvpe("/bin/su", ..., environ)
# 执行 /bin/su，攻击者获得 root shell
```

### 2. TOCTOU 攻击场景

**时间窗口攻击**:

```
时间线:
────────────────────────────────────────────────────────────────────

T1: 用户请求 msmemscope /opt/legitimate_tool
    │
T2: ExecCmd 构造函数执行 realpath()
    │   realpath("/opt/legitimate_tool") → "/opt/tools/legitimate"
    │   path_ = "/opt/tools/legitimate"
    │
T3: ★ 攻击者窗口 ★
    │   攻击者在此时刻替换符号链接：
    │   $ mv /opt/legitimate_tool /tmp/backup
    │   $ ln -s /bin/bash /opt/legitimate_tool
    │
T4: Process::DoLaunch 执行 execvpe()
    │   execvpe("/opt/tools/legitimate", ...)  ← 使用的是 T2 时刻解析的路径
    │
    实际执行的是原始目标，TOCTOU 可能不成功
    │
    但如果攻击者替换的是原始文件：
    │   $ mv /opt/tools/legitimate /tmp/backup
    │   $ ln -s /bin/bash /opt/tools/legitimate
    │
T5: execvpe 执行攻击者的程序
```

**TOCTOU 风险分析**:
- realpath() 解析后立即使用，时间窗口极小
- 但如果 execvpe 使用原始路径而非解析后的路径，存在风险
- 本代码使用 realpath() 解析结果，TOCTOU 风险较低

## 缓解措施建议

### 1. 执行前安全验证（推荐）

**添加路径安全检查**

```cpp
// csrc/framework/process.cpp - 添加验证函数
#include <sys/stat.h>
#include <unistd.h>

enum PathSafetyLevel {
    SAFE,
    WARNING,
    DANGER
};

struct PathSafetyResult {
    PathSafetyLevel level;
    std::string message;
};

PathSafetyResult CheckPathSafety(const std::string& resolvedPath) {
    struct stat st;
    PathSafetyResult result{SAFE, ""};
    
    if (stat(resolvedPath.c_str(), &st) != 0) {
        result.level = DANGER;
        result.message = "Cannot stat target file";
        return result;
    }
    
    // 1. 检查 setuid/setgid 位
    if (st.st_mode & S_ISUID) {
        result.level = DANGER;
        result.message = "Target file has setuid bit - not allowed";
        return result;
    }
    
    if (st.st_mode & S_ISGID) {
        result.level = DANGER;
        result.message = "Target file has setgid bit - not allowed";
        return result;
    }
    
    // 2. 检查属主
    uid_t currentUid = getuid();
    if (st.st_uid != currentUid && st.st_uid != 0) {
        result.level = WARNING;
        result.message = "Target file owned by user " + std::to_string(st.st_uid);
    }
    
    // 3. 检查 world-writable
    if (st.st_mode & S_IWOTH) {
        result.level = DANGER;
        result.message = "Target file is world-writable - security risk";
        return result;
    }
    
    // 4. 检查目录权限（向上遍历）
    std::string dirPath = resolvedPath;
    size_t lastSlash = dirPath.find_last_of('/');
    if (lastSlash != std::string::npos) {
        dirPath = dirPath.substr(0, lastSlash);
        struct stat dirSt;
        if (stat(dirPath.c_str(), &dirSt) == 0) {
            if (dirSt.st_mode & S_IWOTH) {
                result.level = WARNING;
                result.message += " [Parent directory is world-writable]";
            }
        }
    }
    
    return result;
}
```

**修改 ExecCmd 构造函数**

```cpp
ExecCmd::ExecCmd(std::vector<std::string> const &args) : path_{}, argc_{0}, args_{args}
{
    if (args_.empty()) {
        return;
    }

    char *absPath = realpath(args[0].c_str(), nullptr);
    if (absPath) {
        std::string resolvedPath(absPath);
        free(absPath);
        
        // 安全验证
        auto safety = CheckPathSafety(resolvedPath);
        
        if (safety.level == DANGER) {
            LOG_ERROR("Target program validation failed: %s", safety.message.c_str());
            path_ = "";  // 不设置路径，防止执行
            return;
        }
        
        if (safety.level == WARNING) {
            LOG_WARN("Target program warning: %s", safety.message.c_str());
        }
        
        path_ = resolvedPath;
    } else {
        LOG_ERROR("Cannot resolve path: %s", args[0].c_str());
        path_ = args[0];
    }

    argc_ = static_cast<int>(args.size());
    for (auto &arg : args_) {
        argv_.push_back(const_cast<char *>(arg.data()));
    }
    argv_.push_back(nullptr);
}
```

## 风险评级总结

### 最终风险评估

| 维度 | 评分 | 说明 |
|------|------|------|
| 可达性 | 30/30 | realpath() 正常工作，符号链接可被利用 |
| 可控性 | 25/30 | 用户可控制目标路径 |
| 缓解措施 | -5 | realpath() 有基本解析，但无权限验证 |
| **总置信度** | **85** | 真实安全风险（需特权运行场景） |

### 风险降级因素

| 因素 | 影响 |
|------|------|
| 工具设计意图 | 用户需要指定目标程序，这是核心功能 |
| realpath() 正确解析 | 符号链接被正确解析，不会导致 TOCTOU |
| 非特权运行 | 如果工具无特权运行，风险为低 |

### 结论

**这是一个路径安全验证缺失问题。**

- **设计层面**: 工具需要执行用户指定的程序，这是功能需求
- **安全层面**: 缺少对目标程序的安全验证，存在权限提升风险

**风险判断**:
- 如果工具以普通用户权限运行 → **低风险，设计意图**
- 如果工具以特权运行 → **中高风险，需要安全验证**

**建议**: 
实施路径安全检查，特别是：
1. 检查 setuid/setgid 位
2. 检查文件属主
3. 检查 world-writable 权限
4. 添加用户确认机制
