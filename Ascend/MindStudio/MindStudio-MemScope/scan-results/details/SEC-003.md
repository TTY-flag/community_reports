### 2.1 漏洞代码位置

**文件**: `csrc/framework/process.cpp`
```cpp

void Process::DoLaunch(const ExecCmd &cmd) const
{
    // pass all env-vars from global variable "environ"
    execvpe(cmd.ExecPath().c_str(), cmd.ExecArgv(), environ);

}
```

### 2.2 ExecCmd构造函数中的"保护"机制

**文件**: `csrc/framework/process.cpp:42-63`
```cpp
ExecCmd::ExecCmd(std::vector<std::string> const &args) : path_{}, argc_{0}, args_{args}
{
    if (args_.empty()) {
        return;
    }

    /// filename to absolute path
    char *absPath = realpath(args[0].c_str(), nullptr);  // 唯一的"验证"
    if (absPath) {
        path_ = std::string(absPath);
        free(absPath);
        absPath = nullptr;
    } else {
        path_ = args[0];  // realpath失败时直接使用原始路径
    }

    argc_ = static_cast<int>(args.size());
    for (auto &arg : args_) {
        argv_.push_back(const_cast<char *>(arg.data()));
    }
    argv_.push_back(nullptr);
}
```

### 2.3 漏洞根因分析

| 防护措施 | 实现状态 | 安全评估 |
|----------|----------|----------|
| `realpath()` 路径规范化 | ✅ 已实现 | ⚠️ **不足** - 仅解析符号链接和相对路径，无安全边界 |
| 白名单验证 | ❌ 未实现 | 攻击者可执行任意路径的可执行文件 |
| 所有权/权限验证 | ❌ 未实现 | 攻击者可执行任意用户拥有的可执行文件 |
| 路径穿越检测 | ❌ 未实现 | realpath()会解析`../`但无法阻止恶意路径 |
| 可执行文件类型验证 | ❌ 未实现 | 未检查文件是否为有效的ELF/脚本 |

**关键问题**: `realpath()` 的设计目的是路径规范化，而非安全验证。它：
1. 不限制可执行文件的来源目录
2. 不验证文件所有者
3. 不检查文件权限（是否具有执行权限）
4. 在路径解析失败时，直接使用原始路径继续执行

### 3.2 攻击向量分析

#### 攻击向量 1: 直接执行任意程序
```bash
# 攻击者可执行任意系统程序
msmemscope /bin/bash -c "malicious_command"

# 执行攻击者控制的程序
msmemscope /tmp/malicious_binary

# 使用相对路径
msmemscope ../../../usr/bin/vim -c "!malicious_command"
```

#### 攻击向量 2: 符号链接攻击
```bash
# 创建指向敏感程序的符号链接
ln -s /usr/bin/sudo /tmp/fake_tool
msmemscope /tmp/fake_tool  # realpath()会解析到/usr/bin/sudo，但执行无保护

# 符号链接指向攻击者控制的内容
ln -s /home/attacker/evil_script.sh /tmp/legitimate_name
msmemscope /tmp/legitimate_name
```

#### 攻击向量 3: 参数注入
```bash
# 传递危险参数给被分析的程序
msmemscope /usr/bin/python3 -c "import os; os.system('malicious')"

# 利用环境变量
msmemscope /usr/bin/env -S "LD_PRELOAD=/tmp/evil.so /bin/bash"
```

### 3.3 环境变量风险

**文件**: `process.cpp:174-223` - `SetPreloadEnv()` 函数

程序在执行目标程序前会设置 `LD_PRELOAD` 环境变量来注入hook库：
```cpp
void Process::SetPreloadEnv()
{
    std::vector<string> hookLibNames{
        "libleaks_ascend_hal_hook.so",
        "libascend_mstx_hook.so",
        "libascend_kernel_hook.so"
    };
    // ...
    setenv(envName.c_str(), preloadEnv.c_str(), 1);  // 第223行
}
```

攻击者可以利用这一点：
```bash
# 如果能控制hook库的位置，可以注入任意代码
msmemscope /path/to/target  # 会加载LD_PRELOAD指定的库
```

## 4. PoC 构建思路

### 4.1 攻击场景说明

**注意**: 本报告不提供完整可执行的PoC代码，仅说明攻击思路用于安全评估。

### 4.2 攻击思路

#### 场景 A: 特权提升
当 MemScope 工具以更高权限运行时（如 sudo 或通过 setuid），攻击者可以：
1. 指定任意程序路径执行
2. 被执行的程序继承 MemScope 的权限
3. 结合 `LD_PRELOAD` 环境变量注入恶意代码

#### 场景 B: 安全边界绕过
如果部署环境中有安全策略限制可执行文件：
1. MemScope 本身可能是允许执行的程序
2. 通过 MemScope 作为跳板执行其他程序
3. 绕过白名单/沙箱限制

#### 场景 C: 供应链攻击
1. 攻击者替换或创建一个同名的恶意可执行文件
2. 路径解析过程中可能被劫持
3. 无签名/校验和验证机制

## 6. 修复建议

### 6.1 短期修复方案 (紧急补丁)

**位置**: `csrc/framework/process.cpp`

```cpp
#include <sys/stat.h>
#include <unistd.h>
#include <fstream>
#include <set>

// 新增：允许执行的可执行文件白名单
static const std::set<std::string> kAllowedBinaries = {
    "/usr/bin/python3",
    "/usr/bin/python",
    // 根据实际需求添加允许的程序
};

// 新增：路径验证函数
static bool ValidateExecutablePath(const std::string& path, std::string& resolvedPath) {
    // 1. 解析绝对路径
    char* absPath = realpath(path.c_str(), nullptr);
    if (!absPath) {
        LOG_ERROR("Failed to resolve path: %s", path.c_str());
        return false;
    }
    resolvedPath = std::string(absPath);
    free(absPath);
    
    // 2. 检查是否在白名单中
    if (kAllowedBinaries.find(resolvedPath) == kAllowedBinaries.end()) {
        LOG_ERROR("Executable not in allowed list: %s", resolvedPath.c_str());
        return false;
    }
    
    // 3. 验证文件所有权和权限
    struct stat fileStat;
    if (stat(resolvedPath.c_str(), &fileStat) != 0) {
        LOG_ERROR("Failed to stat file: %s", resolvedPath.c_str());
        return false;
    }
    
    // 检查是否为常规文件
    if (!S_ISREG(fileStat.st_mode)) {
        LOG_ERROR("Path is not a regular file: %s", resolvedPath.c_str());
        return false;
    }
    
    // 检查是否具有执行权限
    if (!(fileStat.st_mode & (S_IXUSR | S_IXGRP | S_IXOTH))) {
        LOG_ERROR("File is not executable: %s", resolvedPath.c_str());
        return false;
    }
    
    // 4. 检查文件所有者（可选：只允许root或当前用户拥有的文件）
    uid_t currentUid = getuid();
    if (fileStat.st_uid != 0 && fileStat.st_uid != currentUid) {
        LOG_ERROR("File ownership mismatch: %s (owner: %d, current: %d)", 
                  resolvedPath.c_str(), fileStat.st_uid, currentUid);
        return false;
    }
    
    // 5. 检查是否包含符号链接（防止符号链接攻击）
    // realpath已解析，但可以额外检查原路径是否为符号链接
    if (lstat(path.c_str(), &fileStat) == 0 && S_ISLNK(fileStat.st_mode)) {
        LOG_WARN("Symbolic link detected: %s -> %s", path.c_str(), resolvedPath.c_str());
        // 可选：拒绝符号链接
        // return false;
    }
    
    return true;
}

// 修改 ExecCmd 构造函数
ExecCmd::ExecCmd(std::vector<std::string> const &args) : path_{}, argc_{0}, args_{args}
{
    if (args_.empty()) {
        return;
    }

    /// filename to absolute path with validation
    std::string resolvedPath;
    if (!ValidateExecutablePath(args[0], resolvedPath)) {
        // 验证失败，抛出异常或设置错误状态
        LOG_ERROR("Security: Invalid executable path rejected: %s", args[0].c_str());
        throw std::runtime_error("Invalid executable path");
    }
    
    path_ = resolvedPath;
    argc_ = static_cast<int>(args.size());
    for (auto &arg : args_) {
        argv_.push_back(const_cast<char *>(arg.data()));
    }
    argv_.push_back(nullptr);
}
```

```cpp
// 新增：程序路径预验证
bool ValidateUserBinary(const std::vector<std::string>& cmd) {
    if (cmd.empty()) {
        return false;
    }
    
    const std::string& programPath = cmd[0];
    
    // 1. 检查路径是否包含危险字符
    if (programPath.find('\0') != std::string::npos ||
        programPath.find("..") != std::string::npos) {
        LOG_ERROR("Invalid characters in program path");
        return false;
    }
    
    // 2. 检查路径长度
    if (programPath.length() > PATH_MAX) {
        LOG_ERROR("Program path too long");
        return false;
    }
    
    // 3. 如果是相对路径，转换为绝对路径
    if (programPath[0] != '/') {
        char cwd[PATH_MAX];
        if (getcwd(cwd, sizeof(cwd)) == nullptr) {
            LOG_ERROR("Failed to get current directory");
            return false;
        }
        // 后续会在ExecCmd中进一步验证
    }
    
    return true;
}

// 修改 Parse 函数
UserCommand ClientParser::Parse(int32_t argc, char **argv)
{
    UserCommand userCommand;
    InitialConfig(userCommand.config);
    // ... 现有选项解析代码 ...
```

## 7. 总结

### 7.1 漏洞严重性评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **攻击复杂度** | 低 | 直接通过命令行参数触发 |
| **攻击向量** | 本地 | 需要本地访问权限 |
| **权限要求** | 低 | 仅需普通用户权限 |
| **用户交互** | 无 | 不需要其他用户交互 |
| **影响范围** | 完全控制 | 可执行任意代码 |

### 7.2 核心问题

1. **信任边界缺失**: 用户输入直接传递到系统执行函数，无安全验证
2. **缓解措施不足**: `realpath()` 仅提供路径规范化，不提供安全边界
3. **攻击面暴露**: 命令行参数是直接可控的攻击入口

### 7.3 建议行动

1. **立即**: 实施短期修复方案，添加白名单和所有权验证
2. **短期**: 添加全面的输入验证和审计日志
3. **中期**: 实施沙箱隔离和能力控制
4. **长期**: 重构架构，实施最小权限原则
