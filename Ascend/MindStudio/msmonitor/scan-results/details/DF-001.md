## 漏洞描述

Dynolog守护进程在处理RPC请求时，将完整的请求消息内容记录到INFO级别的日志中。攻击者通过RPC端口1778发送包含敏感数据的请求，可能导致配置信息、进程ID列表等敏感数据被持久化存储到日志文件 `/var/log/dynolog.log`，造成信息泄露风险。

**漏洞代码片段**:
```cpp
// SimpleJsonServer.cpp:314-322
void SimpleJsonServerBase::processOne() noexcept
{
    LOG(INFO) << "Waiting for connection.";
    ClientSocketWrapper client;
    if (!client.accept(sock_fd_, ctx_)) {
        return;
    }
    std::string request_str = client.get_message();
    LOG(INFO) << "RPC message received = " << request_str;  // 漏洞点
    auto response_str = processOneImpl(request_str);
    ...
}
```

## 敏感数据类型分析

### RPC请求JSON结构

攻击者发送的RPC请求包含以下敏感字段：

```json
{
    "fn": "setKinetOnDemandRequest",
    "config": "ACTIVITIES_LOG_FILE=/tmp/trace.json\nPROFILE_START_STEP=10\n...",
    "job_id": 12345,
    "pids": [1001, 1002, 1003],
    "process_limit": 100
}
```

### 敏感字段详细说明

| 字段 | 敏感性 | 说明 |
|------|--------|------|
| `config` | **高敏感** | 包含完整的性能采集配置，包括输出文件路径、采集参数等 |
| `pids` | **中敏感** | 目标进程ID列表，暴露系统进程信息 |
| `job_id` | **中敏感** | 作业标识符，可能暴露业务信息 |

### config字段详细内容

`config`字段包含以下配置项（完整记录到日志）：

## 触发条件

### 前提条件

1. **服务运行状态**: Dynolog守护进程正在运行并监听TCP端口1778
2. **网络可达性**: 攻击者能够连接到RPC端口（IPv6监听，`in6addr_any`）
3. **认证绕过**:
   - 默认模式（`NO_CERTS_MODE`）：无需认证，任何客户端可直接连接
   - TLS模式：需要客户端证书，但证书验证可能存在弱点
4. **日志配置**: INFO级别日志默认开启（`FLAGS_logtostderr = 1`）

### 触发流程

```
Step 1: 攻击者建立TCP连接到目标服务器端口1778
        │
        ▼
Step 2: 发送4字节长度前缀 + JSON消息体
        │   msg_size = strlen(json_request)
        │   socket.write(msg_size)
        │   socket.write(json_request)
        ▼
Step 3: 服务端 SimpleJsonServer::processOne() 接收连接
        │
        ▼
Step 4: get_message() 读取消息内容到 request_str
        │
        ▼
Step 5: LOG(INFO) 将完整 request_str 写入日志文件
        │   ← 漏洞触发点
        ▼
Step 6: 敏感数据持久化存储到 /var/log/dynolog.log
```

## 攻击向量分析

### 攻击向量 1: 信息收集攻击

**攻击目标**: 收集系统配置信息和进程信息

**攻击步骤**:
1. 连接到目标服务器端口1778
2. 发送包含目标进程ID的RPC请求
3. 查看日志文件获取完整的配置信息

**攻击示例代码** (Python):
```python
import socket
import json
import struct

def exploit_info_leakage(target_ip, port=1778):
    # 构造包含敏感数据的请求
    request = {
        "fn": "setKinetOnDemandRequest",
        "config": "ACTIVITIES_LOG_FILE=/sensitive/path/data.json\\nPROFILE_START_STEP=1",
        "job_id": 999,
        "pids": [1234, 5678],  # 目标进程ID
        "process_limit": 100
    }
    
    json_str = json.dumps(request)
    msg_size = struct.pack('<i', len(json_str))
    
    # 发送请求
    sock = socket.socket(socket.AF_INET6, socket.SOCK_STREAM)
    sock.connect((target_ip, port))
    sock.send(msg_size + json_str.encode())
    
    # 请求已记录到日志，攻击者可通过其他途径访问日志文件
    # 或等待管理员查看日志时发现敏感信息
    
    sock.close()
```

### 攻击向量 2: 日志污染攻击

**攻击目标**: 污染日志文件，注入恶意内容或填充大量数据

**攻击步骤**:
1. 构造包含大量垃圾数据的config字段
2. 发送多个请求填充日志文件
3. 导致日志文件快速膨胀或包含恶意内容

**攻击示例**:
```python
def exploit_log_pollution(target_ip, port=1778):
    # 注入大量数据填充日志文件

    for i in range(100):
        send_rpc_request(target_ip, port, request)
```

### 攻击向量 3: 社会工程学辅助攻击

**攻击目标**: 通过日志记录辅助后续攻击

**攻击场景**:
1. 攻击者发送包含特定标记的请求
2. 请求被记录到日志文件
3. 后续攻击者可利用日志中的信息：
   - 确认服务正在运行
   - 获取进程信息用于针对性攻击
   - 获取配置路径用于文件访问攻击

## 利用场景

### 场景 1: 内部威胁场景

**攻击者**: 内部具有低权限账户的用户

**攻击路径**:
```
1. 用户连接到RPC端口1778（同一网络）
2. 发送包含敏感进程信息的请求
3. 利用低权限访问 /var/log/dynolog.log（假设日志权限配置不当）
4. 获取其他用户的进程配置信息
```

**影响**: 突破权限边界，获取本不应看到的配置信息

### 场景 2: 外部攻击场景（需要认证绕过）

**攻击者**: 外部远程攻击者

**前提条件**: 
- TLS认证配置为 `NO_CERTS_MODE`（无认证）
- 或客户端证书验证存在缺陷

**攻击路径**:
```
1. 远程连接到目标服务器端口1778
2. 无需认证直接发送RPC请求
3. 请求内容被记录到服务器日志
4. 通过其他漏洞（如文件遍历）访问日志文件
```

### 场景 3: 多阶段攻击链

**攻击流程**:
```
Phase 1: 信息探测
         发送getStatus/getVersion请求，确认服务状态
         ↓
Phase 2: 配置收集
         发送setKinetOnDemandRequest，记录配置信息到日志
         ↓
Phase 3: 日志分析
         通过其他途径访问日志文件（权限提升漏洞、配置错误等）
         ↓
Phase 4: 针对性攻击
         利用收集的配置信息进行针对性攻击
```

## 影响评估

### 直接影响

| 影响类型 | 严重程度 | 说明 |
|----------|----------|------|
| 配置信息泄露 | High | 性能采集配置、文件路径等敏感信息被记录 |
| 进程信息泄露 | Medium | 目标进程ID列表暴露系统进程信息 |
| 业务信息泄露 | Medium | job_id可能暴露业务作业信息 |
| 日志污染 | Medium | 攻击者可注入恶意内容到日志文件 |

### 间接影响

1. **后续攻击辅助**: 日志中的信息可能被用于后续攻击的情报收集
2. **日志完整性破坏**: 注入的恶意内容可能干扰日志审计和分析
3. **合规性问题**: 敏感数据记录到日志可能违反数据保护规定

### 可能泄露的具体数据

- **文件路径**: `ACTIVITIES_LOG_FILE` 参数暴露输出文件存储位置
- **设备配置**: `PROFILE_ACTIVITIES` 参数暴露使用的设备类型
- **进程信息**: `pids` 数组暴露目标进程ID
- **业务标识**: `job_id` 暴露作业/任务标识符

## 修复建议

### 建议方案 1: 移除敏感数据日志记录（推荐）

**修改位置**: SimpleJsonServer.cpp:322

**修改方案**:
```cpp
// 原代码（漏洞点）
LOG(INFO) << "RPC message received = " << request_str;

// 修复方案：移除完整请求记录
LOG(INFO) << "RPC message received from client";
// 或仅记录请求类型（不含敏感内容）
LOG(INFO) << "RPC request function: " << /* 从request_str提取fn字段 */;
```

**优点**: 
- 完全消除信息泄露风险
- 保留基本日志审计能力
- 最小化代码改动

### 建议方案 2: 数据脱敏处理

**修改方案**: 在记录日志前对敏感字段进行脱敏

```cpp
void SimpleJsonServerBase::processOne() noexcept
{
    LOG(INFO) << "Waiting for connection.";
    ClientSocketWrapper client;
    if (!client.accept(sock_fd_, ctx_)) {
        return;
    }
    std::string request_str = client.get_message();
    
    // 脱敏处理：移除config和pids字段内容
    std::string sanitized_request = sanitizeRequest(request_str);
    LOG(INFO) << "RPC message received = " << sanitized_request;
    
    auto response_str = processOneImpl(request_str);
    ...
}

std::string sanitizeRequest(const std::string& request) {
    // 解析JSON并移除敏感字段
    try {
        auto json = nlohmann::json::parse(request);
        if (json.contains("config")) {
            json["config"] = "[REDACTED]";
        }
        if (json.contains("pids")) {
            json["pids"] = "[REDACTED]";
        }
        return json.dump();
    } catch (...) {
        return "[UNPARSEABLE REQUEST]";
    }
}
```

### 建议方案 3: 降低日志级别

**修改方案**: 将敏感数据的日志记录降级为DEBUG或更高级别

```cpp
// 使用DEBUG级别（需要显式配置才会记录）
LOG(DEBUG) << "RPC message received = " << request_str;

// 或使用自定义级别
LOG_IF(INFO, FLAGS_verbose_rpc_logging) << "RPC message received = " << request_str;
```

**配合配置**: 在生产环境中禁用verbose日志记录

### 建议方案 4: 完整安全审计方案

**综合修复措施**:

1. **移除完整请求日志**: 采用方案1或方案2
2. **添加敏感数据检测**: 检测并阻止包含明显敏感数据的请求
3. **日志访问控制**: 限制日志文件访问权限（chmod 600）
4. **日志加密**: 对日志文件内容进行加密存储

### 修复优先级

| 方案 | 优先级 | 难度 | 安全收益 |
|------|--------|------|----------|
| 方案1: 移除完整日志 | **高** | 低 | 高 |
| 方案2: 数据脱敏 | 中 | 中 | 高 |
| 方案3: 降低日志级别 | 低 | 低 | 中 |
| 方案4: 综合方案 | 高 | 高 | 最高 |

**推荐实施**: 优先实施方案1，后续考虑方案4的完整审计措施
