# VULN-SEC-XMOD-001: 权限提升视角分析

## 漏洞关联声明

**此漏洞与 [cross-module-configs-to-command-exec](./cross-module-configs-to-command-exec.md) 是同一技术缺陷的不同分析视角。**

| 属性 | cross-module-configs-to-command-exec | VULN-SEC-XMOD-001 |
|------|--------------------------------------|-------------------|
| **漏洞ID** | cross-module-configs-to-command-exec | VULN-SEC-XMOD-001 |
| **主要CWE** | CWE-94 (代码注入), CWE-78 (命令注入) | CWE-269 (权限管理不当) |
| **分析视角** | 攻击链与代码执行 | 权限继承与影响范围 |
| **攻击入口** | `@autotune(configs=...)` | 同一入口 |
| **根本缺陷** | 输入验证缺失 | 同一缺陷 |
| **修复方案** | 输入内容验证 + 白名单机制 | 同一方案 |

---

## 判定：合并分析

**原因**：

1. **代码位置完全相同**：两者都指向 `tuner.py:251-280` 的 `autotune` 装饰器

2. **攻击链完全相同**：
   ```
   用户配置 → autotune 装饰器 → kernel_modifier 替换源码 → 
   compiler 编译 → driver 动态加载 → 代码执行
   ```

3. **"权限提升"实为"权限继承"**：
   - CWE-269 适用于权限管理不当导致的实际权限提升
   - 本漏洞中，恶意代码以 mskl 库进程身份运行，是**继承**当前权限，并非提升
   - 这只是代码执行的直接后果，不是独立的权限管理缺陷

4. **修复方案一致**：加强输入验证即可同时解决两个视角描述的问题

## 权限影响补充分析

虽然与主报告是同一漏洞，但"权限视角"揭示了重要的攻击后果：

### 恶意代码获得的能力

当攻击者通过 `@autotune` 注入的代码在 `.so` 模块加载时执行，可获得以下权限：

| 权限类型 | 具体能力 | 风险等级 |
|----------|----------|----------|
| **进程权限** | 以 mskl 库进程身份运行，继承所有文件系统访问权限 | 高 |
| **NPU 设备访问** | 通过 `acl.rt` 接口访问 NPU 硬件 | 高 |
| **NPU 内存操作** | 调用 `driver.malloc/free/memcpy` 操作设备内存 | 高 |
| **CANN 库调用** | 直接调用 CANN 库函数（如 `aclInit`, `rtSetDevice`） | 高 |
| **数据访问** | 访问内核参数中的张量数据（可能包含敏感训练数据） | 高 |

### 多租户环境风险

在 AI 训练平台/共享算力环境中：

```
[恶意用户] --提供恶意 configs--> [autotune 装饰器]
                                      ↓
                              [恶意 .so 模块加载]
                                      ↓
                              [以平台权限执行]
                                      ↓
                    [访问其他用户数据 / 平台敏感资源]
```

### 凭证传递分析

```
┌─────────────────────────────────────────────────────────────┐
│                    跨模块凭证传递链                           │
├─────────────────────────────────────────────────────────────┤
│ 用户配置值 (configs)                                        │
│     ↓ [无内容验证]                                          │
│ 源代码内容 (kernel_modifier._replace_param)                 │
│     ↓ [无签名验证]                                          │
│ 编译产物 (.so 文件)                                          │
│     ↓ [无代码完整性验证]                                     │
│ 可执行代码 (driver.importlib.exec_module)                   │
│     ↓                                                       │
│ 【获得 mskl 库进程权限】                                      │
└─────────────────────────────────────────────────────────────┘
```

每个环节都**缺失安全验证**，形成完整的凭证传递链。
