# 9b345463d9b3：compile_executable不受信任构建脚本致任意代码执行

## 漏洞概述

| 属性 | 值 |
|------|-----|
| **漏洞ID** | 9b345463d9b3 |
| **漏洞类型** | Untrusted Code Execution (关联 CWE-78: OS Command Injection, CWE-94: Code Injection) |
| **严重程度** | High → Critical (取决于部署场景) |
| **置信度** | 90% → **已确认** |
| **影响版本** | MindStudio-Kernel-Launcher (当前版本) |
| **攻击复杂度** | 低 |
| **影响范围** | 任意代码执行 |

### 漏洞摘要

MindStudio-Kernel-Launcher 的 `compile()` 和 `compile_executable()` 函数接收用户提供的 `build_script` 路径，并通过 `subprocess.run(['bash', build_script, ...])` 执行。虽然 `FileChecker` 对脚本文件进行了元数据验证（存在性、权限、大小等），但**完全不验证脚本内容**，导致攻击者可以提供恶意脚本并在目标环境执行任意命令。

## 与已报告漏洞的区别

### cross-module-configs-to-command-exec.md

该漏洞通过 `@autotune(configs=...)` 参数注入代码：

```
用户输入 configs → C++ 源码注入 → 编译 → 动态加载执行
```

**特点**：需要利用 autotune 装饰器，通过 configs 参数注入。

### 本漏洞 (9b345463d9b3)

直接提供恶意 `build_script`：

```
用户控制 build_script → FileChecker 检查(通过) → bash 执行
```

**特点**：直接执行，不依赖代码注入，更简单直接。

### 攻击面对比

| 维度 | cross-module 漏洞 | 本漏洞 |
|------|-------------------|--------|
| 攻击入口 | `autotune(configs=...)` | `compile(build_script=...)` |
| 攻击路径 | configs → C++ 源码 → 编译 → 加载 | script → bash 直接执行 |
| 复杂度 | 中等（需要构造有效 C++ 代码） | 低（任意 shell 命令） |
| 依赖条件 | 需要编译环境和内核源文件 | 仅需脚本文件 |
| 防护绕过 | 无输入验证 | FileChecker 元数据检查 |

## 修复建议

### 1. 短期修复：内容验证（优先级：高）

在 `FileChecker` 中添加脚本内容安全检查：

```python
import re

class FileChecker:
    # 危险命令模式
    DANGEROUS_PATTERNS = [
        r'\brm\s+-rf\b',           # 危险删除命令
        r'\bchmod\s+[0-7]{3,4}\b',  # 权限修改
        r'\bchown\b',              # 所有者修改
        r'\bcurl\s+.*\|\s*bash',   # 远程脚本执行
        r'\bwget\s+.*\|\s*bash',   # 远程脚本执行
        r'\bbash\s+-i\b',          # 反向 shell
        r'\bnc\s+-[elp]',          # netcat 反向 shell
        r'/dev/tcp/',              # bash 网络重定向
        r'\bexport\b.*\bPATH\b',   # PATH 篡改
        r'\beval\b',               # 动态代码执行
        r'\bexec\b',               # 进程替换
        r'\bsource\b.*http',       # 远程脚本加载
    ]
    
    def check_script_content(self):
        """检查脚本内容是否包含危险命令"""
        if self.file_type not in ["file", "app"]:
            return True
        
        try:
            with open(self.absolute_path, 'r', encoding='utf-8', errors='ignore') as f:
                content = f.read()
            
            for pattern in self.DANGEROUS_PATTERNS:
                if re.search(pattern, content, re.IGNORECASE):
                    logger.error(f"Script contains dangerous pattern: {pattern}")
                    return False
            return True
        except Exception as e:
            logger.error(f"Failed to read script: {e}")
            return False
    
    def check_input_file(self):
        # ... 现有检查 ...
        
        # 新增：脚本内容检查
        if self.file_type in ["file", "app"] and self.absolute_path.endswith('.sh'):
            if not self.check_script_content():
                return False
        
        return True
```

### 2. 中期修复：白名单机制（优先级：高）

实现构建命令白名单：

```python
class BuildScriptValidator:
    ALLOWED_COMMANDS = {
        'g++', 'gcc', 'clang++', 'clang',  # 编译器
        'ar', 'ld', 'nm', 'objcopy',       # 工具链
        'mkdir', 'cp', 'mv',               # 文件操作
    }
    
    FORBIDDEN_COMMANDS = {
        'bash', 'sh', 'zsh', 'python', 'perl', 'ruby',  # 解释器
        'curl', 'wget', 'nc', 'telnet',                  # 网络工具
        'ssh', 'scp', 'rsync',                            # 远程访问
        'crontab', 'at', 'systemctl', 'service',         # 系统服务
        'sudo', 'su', 'doas',                             # 权限提升
    }
    
    @classmethod
    def validate(cls, script_path: str) -> bool:
        """验证构建脚本只包含允许的命令"""
        with open(script_path, 'r') as f:
            content = f.read()
        
        # 解析脚本中的命令
        for line in content.split('\n'):
            line = line.strip()
            if not line or line.startswith('#'):
                continue
            
            # 提取命令
            cmd = line.split()[0] if line.split() else ''
            cmd = os.path.basename(cmd)
            
            if cmd in cls.FORBIDDEN_COMMANDS:
                raise SecurityError(f"Forbidden command: {cmd}")
        
        return True
```

```python
import subprocess
import docker  # 或使用其他沙箱技术

def compile_with_sandbox(build_script: str, src_path: str, output_path: str):
```

```python
    }
    
    @classmethod
    def compile(cls, template_name: str, src: str, output: str, flags: list = None):
        if template_name not in cls.TEMPLATES:
            raise ValueError(f"Unknown template: {template_name}")
        
        # 使用预定义模板，避免用户控制命令
        cmd_str = cls.TEMPLATES[template_name].format(
            src=src,
            output=output,
            flags=' '.join(flags or [])
        )
        
        # 安全分割命令
        cmd = shlex.split(cmd_str)
        result = subprocess.run(cmd, capture_output=True, text=True)
        return result
```

### 5. 加固 FileChecker（优先级：中）

将警告改为阻止：

```python
def check_input_file(self):
    # ... 现有检查 ...
    
    # 软链接检查：警告 → 阻止
    if self.is_soft_link_recusively():
        logger.error(f"Path:{self.absolute_path} contains soft link, rejected for security.")
        return False  # 改为返回 False
    
    # 所有者检查：警告 → 阻止
    file_stat = os.stat(self.absolute_path)
    if uid != 0 and uid != file_stat.st_uid and file_stat.st_uid != 0:
        logger.error(f"Path:{self.absolute_path} is not owned by the current user, rejected.")
        return False  # 改为返回 False
    
    # 组/其他用户写权限检查：警告 → 阻止
    if (group_permission & os.W_OK) != 0 or (other_permission & os.W_OK) != 0:
        logger.error(f"Path:{self.absolute_path} has group/other write permission, rejected.")
        return False  # 改为返回 False
    
    return True
```

## 总结

### 漏洞确认

这是一个**真实的安全风险**，虽然不是传统意义上的 shell 命令注入，但存在以下问题：

1. **信任边界不清**：用户提供的脚本被完全信任执行
2. **验证不充分**：FileChecker 只验证元数据，不验证内容
3. **可利用场景明确**：多租户平台、共享环境、供应链攻击
4. **影响严重**：任意代码执行，可能导致数据泄露、权限提升

### 修复优先级

| 优先级 | 修复方案 | 预期效果 |
|--------|----------|----------|
| P0 | 禁止自定义脚本，使用预定义模板 | 彻底消除风险 |
| P1 | 沙箱隔离执行 | 限制执行影响范围 |
| P1 | 脚本内容白名单验证 | 阻止危险命令 |
| P2 | FileChecker 警告改阻止 | 减少绕过可能 |
| P3 | 文档安全警告 | 提高开发者意识 |

### 相关 CWE 参考

- **CWE-78**: OS Command Injection
- **CWE-94**: Improper Control of Generation of Code ('Code Injection')
- **CWE-732**: Incorrect Permission Assignment for Critical Resource
- **CWE-250**: Execution with Unnecessary Privileges
- **CWE-863**: Incorrect Authorization
