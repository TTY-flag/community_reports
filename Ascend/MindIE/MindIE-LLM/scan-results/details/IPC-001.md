# IPC-001: ParseResponse共享内存messageSize未验证致越界读取

## 漏洞概述

| 属性 | 值 |
|------|-----|
| **漏洞ID** | IPC-001 |
| **类型** | Buffer Overread (缓冲区越界读取) |
| **CWE** | CWE-125: Out-of-bounds Read |
| **严重程度** | HIGH |
| **状态** | VERIFIED (真实漏洞) |
| **受影响文件** | `src/executor/ipc_communicator.cpp` |
| **受影响函数** | `IPCCommunicator::ParseResponse()` |
| **受影响行号** | 249-251 |

### 漏洞描述

在 `IPCCommunicator::ParseResponse()` 函数中，`messageSize` 值直接从共享内存缓冲区读取而未进行边界验证。攻击者若能控制共享内存内容，可设置一个超大 `messageSize` 值，导致 `ParseFromArray()` 读取超出共享内存缓冲区边界，造成：

- **信息泄露**: 读取相邻内存区域数据
- **拒绝服务**: 访问无效内存导致进程崩溃
- **潜在内存损坏**: 取决于 protobuf 解析实现

## 触发条件分析

### 代码路径分析

**漏洞代码 (ipc_communicator.cpp:247-266):**
```cpp
bool IPCCommunicator::ParseResponse(ExecuteResponse &executeResponse, char *sharedBuf) const
{
    uint32_t messageSize = *reinterpret_cast<uint32_t *>(sharedBuf);  // [1] 无验证读取
    auto spanDeserialize = PROF(INFO, Domain("Executor").SpanStart("deserializeResponses"));
    if (!executeResponse.ParseFromArray(sharedBuf + sizeof(uint32_t), messageSize)) {  // [2] 使用未验证值
        MINDIE_LLM_LOG_ERROR("Failed to deserialize buffer.");
        PROF(spanDeserialize.SpanEnd());
        return false;
    }
    // ...
}
```

**[1] 问题点**: `messageSize` 直接从共享内存的前4字节读取，无任何边界检查
**[2] 问题点**: `ParseFromArray()` 使用未验证的 `messageSize` 读取数据

### 触发条件

| 条件 | 可达性 | 说明 |
|------|--------|------|
| 共享内存可写 | **部分可达** | 需要获得对共享内存段的写权限 |
| 信任边界跨越 | **存在** | IPC通信涉及跨进程数据交换 |
| 输入验证缺失 | **确认** | Read路径完全没有边界检查 |

### 已存在的缓解措施

1. **权限控制** (部分有效):
   - 共享内存创建时设置为 0600 权限 (仅owner可读写)
   - `SharedMemoryUIDAndPermissionChecker()` 检查 UID 和权限
   - 信号量也有类似权限检查 (`CheckSemaphoreOwnerAndPermission()`)

2. **Write路径有边界检查** (不缓解Read路径):
   ```cpp
   // shared_memory.cpp:169
   if (dstOffset + size > this->mCurSize) {
       MINDIE_LLM_LOG_ERROR("No enough shared memory...");
       return false;
   }
   ```

3. **辅助函数存在但未使用**:
   - `GetBufEnd()` 方法可返回缓冲区结束位置，但 `ParseResponse` 未使用

### 绕过缓解的可能性

| 攻击场景 | 可行性 | 说明 |
|----------|--------|------|
| 同UID进程攻击 | **高** | 同用户的其他进程可访问共享内存 |
| 子进程恶意行为 | **中高** | 子进程可能继承或获取访问权限 |
| 权限提升后攻击 | **中** | 需先获得权限提升 |
| 信号量竞态 | **中** | 信号量同步可能被绕过 |

## PoC 构思 (概念性)

**注意**: 不提供完整可执行 PoC，仅描述攻击思路

### 攻击前置条件
1. 攻击者需为同一 UID 用户或已获得权限提升
2. 能访问目标共享内存段 (如 `/dev/shm/<name>_response`)

### 攻击步骤概念
1. **定位共享内存**: 通过 `/dev/shm/` 目录找到目标进程的响应共享内存
2. **获取访问权**: 使用 `shm_open()` 映射共享内存 (需同UID)
3. **监控时机**: 观察信号量状态，等待响应写入完成
4. **篡改数据**: 将共享内存前4字节改为超大值 (> 共享内存实际大小)
5. **触发读取**: 目标进程调用 ParseResponse 时触发越界读取

### 危害演示方式
- **信息泄露**: 设置 messageSize 为合理大值，读取相邻进程内存
- **DoS**: 设置 messageSize 为极大值 (如 UINT32_MAX)，触发 SIGSEGV

## 修复建议

### 方案1: 添加边界检查 (推荐)

```cpp
bool IPCCommunicator::ParseResponse(ExecuteResponse &executeResponse, char *sharedBuf) const
{
    uint32_t messageSize = *reinterpret_cast<uint32_t *>(sharedBuf);
    
    // [新增] 边界检查
    size_t availableSize = responseSharedMemory_.sharedMemory->GetBufEnd() - sharedBuf;
    if (messageSize > availableSize - sizeof(uint32_t)) {
        MINDIE_LLM_LOG_ERROR("Invalid message size: " << messageSize 
                             << ", available: " << availableSize - sizeof(uint32_t));
        return false;
    }
    
    auto spanDeserialize = PROF(INFO, Domain("Executor").SpanStart("deserializeResponses"));
    if (!executeResponse.ParseFromArray(sharedBuf + sizeof(uint32_t), messageSize)) {
        // ...
    }
    // ...
}
```

### 方案2: 传递共享内存大小信息

修改函数签名，传递缓冲区大小:

```cpp
bool ParseResponse(ExecuteResponse &executeResponse, char *sharedBuf, size_t bufSize) const;
```

### 方案3: 使用 SharedMemory 类的 Read 方法

新增 Read 方法，封装边界检查:

```cpp
// shared_memory.h - 新增方法
bool Read(uint32_t srcOffset, char *dst, uint32_t size) const;

// shared_memory.cpp - 实现
bool SharedMemory::Read(uint32_t srcOffset, char *dst, uint32_t size) const
{
    if (!valid) { return false; }
    if (srcOffset + size > mCurSize) {
        MINDIE_LLM_LOG_ERROR("Read out of bounds: offset=" << srcOffset << ", size=" << size);
        return false;
    }
    memcpy(dst, mMapBuf + srcOffset, size);
    return true;
}
```

### 方案4: 强化权限隔离

- 考虑使用 SELinux/AppArmor 进一步隔离
- 为关键进程设置专用 UID
- 使用 namespace 容器隔离

## 结论

**IPC-001 是一个真实的、可被利用的缓冲区越界读取漏洞。**

虽然权限缓解措施限制了攻击面，但以下场景仍可利用:
1. 同 UID 进程间的恶意攻击
2. 恶意子进程篡改父进程共享内存
3. 权限提升后针对服务的攻击

建议立即实施边界检查修复，优先采用方案1或方案3。
